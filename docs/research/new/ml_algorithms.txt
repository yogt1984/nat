  1. MOMENTUM CONTINUATION CLASSIFIER

================================================================================

Status:       PROPOSED
Created:      2026-04-04 (design), 2026-06-03 (expanded)
File:         scripts/algorithms/momentum_continuation.py
Signal:       alg_mc_signal
Polarity:     high_long
ML Method:    Logistic Regression (primary) / LightGBM (alternative)
Gate:         Only active in low-entropy regimes (ent_tick_1m_mean < 0.3)

================================================================================
1.1  THESIS
================================================================================

In low-entropy regimes, momentum persists. The order book is dominated by
directional flow, market makers have pulled liquidity, and price moves
autocorrelate over minutes to hours. This is the regime where trend-following
works — and where a classifier can predict whether the current momentum will
continue or reverse.

The edge comes from COMBINING momentum direction with microstructure
confirmation: whale flow alignment, Hurst persistence, and low toxicity.
A simple momentum signal (e.g., momentum > 0 -> long) captures the trend
but produces many false signals during choppy low-entropy transitions. The
classifier learns the difference between "genuine low-entropy trend" and
"temporary entropy dip in a range."

This algorithm is only active when the Entropy-Gated Strategy Switcher
(Algorithm #1 in the design proposal) routes to it. During high-entropy
or uncertain regimes, the signal is zero.

Crypto-native angle:
  Hyperliquid's funding rate mechanism creates periodic forced flows every
  8 hours. When funding is extreme AND entropy is low, the crowd is
  directionally positioned AND the market is trending — a strong setup for
  continuation. This interaction is hard to capture without the classifier.

================================================================================
1.2  SIGNAL CONSTRUCTION
================================================================================

1.2.1  Target variable
---------------------------------------------------------------------

  y_t = sign(r_{t+h})

  where r_{t+h} = (P_{t+h} - P_t) / P_t   is the forward return at
  horizon h.

  h is configurable; default = 100 bars at 5min = 500min (~8.3 hours).
  This matches the 100min horizon used in the existing gauntlet.

  Label encoding:
    y = +1 if r_{t+h} > 0    (price went up)
    y = -1 if r_{t+h} < 0    (price went down)
    y =  0 if |r_{t+h}| < min_return   (filter out noise, default 0.5 bps)

  Bars with y = 0 are excluded from training (dead zone filter).

1.2.2  Feature vector
---------------------------------------------------------------------

  All features are bar-aggregated via aggregate_bars() from
  scripts/cluster_pipeline/preprocess.py using the 5min timeframe.
  Suffix _mean, _std, _last, _slope per the aggregation spec.

  Feature                             Code name                        Weight
  -----------------------------------------------------------------------
  1. Entropy-gated momentum           momentum_interaction             High
     = trend_momentum_300_last * (1 - ent_tick_1m_mean)

     Numerical example:
       trend_momentum_300_last = +0.012  (1.2% return over 300 ticks)
       ent_tick_1m_mean = 0.20           (low entropy)
       momentum_interaction = 0.012 * (1 - 0.20) = 0.012 * 0.80 = 0.0096

     When entropy is low (0.20), 80% of momentum passes through.
     When entropy is high (0.80), only 20% passes — signal attenuated.

  2. Whale-confirmed momentum         whale_confirmation               High
     = whale_net_flow_4h_last * sign(trend_momentum_300_last)

     Positive when whales agree with momentum direction.
     Negative when whales diverge (whale selling during uptrend).

     Numerical example:
       whale_net_flow_4h_last = +2.5e6    ($2.5M net whale buying)
       trend_momentum_300_last = +0.012    (uptrend)
       whale_confirmation = 2.5e6 * sign(0.012) = +2.5e6

       whale_net_flow_4h_last = -1.0e6    ($1M net whale selling)
       trend_momentum_300_last = +0.012    (uptrend)
       whale_confirmation = -1.0e6 * sign(0.012) = -1.0e6  (divergence!)

  3. Hurst exponent                   trend_hurst_300_last             Medium
     H > 0.5 = persistent (trending), H < 0.5 = anti-persistent (reverting)
     H ~ 0.5 = random walk (no edge)

  4. Trend strength                   trend_strength_computed          Medium
     = sqrt(trend_momentum_300_last^2 * trend_r_squared_300_last)

     Combines momentum magnitude with linearity. High value = strong
     directional trend with R^2 confirming it's linear, not noisy.

  5. R-squared                        trend_r_squared_300_last         Medium
     Linear fit quality of price over 300 ticks. High R^2 = clean trend.

  6. Monotonicity                     trend_monotonicity_300_last      Medium
     Fraction of ticks where price moved in the dominant direction.
     Range [0, 1]. > 0.6 = directional, < 0.5 = choppy.

  7. Toxicity (inverse)               toxic_vpin_10_mean               Low
     VPIN measures probability of informed trading. LOW vpin = uninformed
     flow dominates = trend is retail-driven = more likely to continue.

  8. Accumulation score               regime_accumulation_score_last   Low
     From Rust regime detection. High score during accumulation phase
     suggests whale positioning, confirming trend direction.

  9. Funding z-score                  ctx_funding_zscore_last          Low
     Extreme funding in SAME direction as trend = crowd agrees = higher
     continuation probability (but also higher reversal risk when it breaks).

  Total: 9 raw features -> 9 input dimensions after interaction terms.

1.2.3  Model specification — Logistic Regression (primary)
---------------------------------------------------------------------

  Loss function (binary cross-entropy with L2 regularization):

    L(beta) = -(1/N) * sum_{i=1}^{N} [
        y_i * log(sigma(z_i)) + (1 - y_i) * log(1 - sigma(z_i))
    ] + (lambda/2) * ||beta||_2^2

  where:
    z_i = beta_0 + sum_{j=1}^{9} beta_j * x_{ij}
    sigma(z) = 1 / (1 + exp(-z))
    lambda = 1/C = regularization strength (default C = 0.1)
    N = number of training samples
    y_i in {0, 1} (recoded from {-1, +1} for cross-entropy)

  Optimization: L-BFGS (sklearn default for small feature sets).
  Convergence: max_iter = 1000, tol = 1e-6.

  Feature standardization: StandardScaler (zero-mean, unit-variance)
  fitted on training set, applied to test set. Stored via model_io.py.

  Probability calibration:
    P(continuation) = sigma(beta_0 + beta^T * x)

    Trade signal:
      If P > 0.60 AND ent_tick_1m_mean < 0.30   -> signal = +(P - 0.5) * 2
      If P < 0.40 AND ent_tick_1m_mean < 0.30   -> signal = -(0.5 - P) * 2
      Otherwise                                   -> signal = 0.0

    Numerical examples:
      P = 0.72, ent = 0.20  -> signal = +(0.72 - 0.5)*2 = +0.44  (long)
      P = 0.35, ent = 0.15  -> signal = -(0.5 - 0.35)*2 = -0.30  (short)
      P = 0.55, ent = 0.25  -> signal = 0.0  (dead zone, skip)
      P = 0.80, ent = 0.50  -> signal = 0.0  (entropy gate blocks)

  Use when feature interactions are important (validated by comparing
  OOS Sharpe of logistic vs LightGBM on first 14 days of data).

  Objective:      binary (log loss)
  Metric:         auc

  Hyperparameter grid (grid search or Bayesian optimization):

    Parameter           Default   Sweep range
    -----------------------------------------------------------------------
    max_depth           5         [3, 5, 7]
    num_leaves          31        [15, 31, 63]
    learning_rate       0.05      [0.01, 0.05, 0.1]
    n_estimators        200       [100, 200, 400]
    min_child_samples   50        [20, 50, 100]

================================================================================
1.5  TRAINING PROTOCOL
================================================================================

  Walk-forward retraining:

    Train window:      14 days of 5min bars (~4032 bars/symbol)
    Validation window:  3 days of 5min bars (~864 bars/symbol)
    Embargo:           100 bars (500 min) between train and validation
    Retrain frequency: weekly (7 days)
    Symbols:           train on pooled BTC+ETH+SOL (symbol as feature)

  Per retrain cycle:
    1. Load parquet data via aggregate_bars(timeframe="5min")
    2. Compute forward returns, apply dead zone filter
    3. Build feature matrix (9 features), apply StandardScaler
    4. Fit model (logistic or LightGBM)
    5. Evaluate on validation window:
       - Sharpe, win rate, precision, AUC
       - OOS/IS Sharpe ratio >= 0.7 required
    6. If validation passes: save model via save_sklearn_model() or
       save_lightgbm_model() from scripts/utils/model_io.py
    7. If validation fails: keep previous model (staleness protection)

  Training script:
    scripts/train_momentum_continuation.py
    Invoked by adaptive_calibrator nightly or manually via:
      python scripts/train_momentum_continuation.py \
          --train-days 14 --val-days 3 --embargo 100

1.6  IMPLEMENTATION
================================================================================

  File: scripts/algorithms/momentum_continuation.py

    from __future__ import annotations
    import math
    import numpy as np
    from .base import AlgorithmFeature, MicrostructureAlgorithm
    from .registry import register

    @register
    class MomentumContinuation(MicrostructureAlgorithm):
        """Logistic/LightGBM momentum continuation classifier.

        Active only in low-entropy regimes. Predicts whether current
        momentum will continue based on whale confirmation, Hurst
        persistence, and toxicity.

        References:
          Moskowitz, Ooi & Pedersen (2012) — Time Series Momentum
          Jegadeesh & Titman (1993) — Returns to Buying Winners
        """

        def __init__(
            self,
            model_path: str | None = None,
            entropy_threshold: float = 0.30,
            probability_long: float = 0.60,
            probability_short: float = 0.40,
        ):
            self._model = None
            self._scaler = None
            self._entropy_threshold = entropy_threshold
            self._p_long = probability_long
            self._p_short = probability_short
            self._tick_count = 0

            if model_path:
                from scripts.utils.model_io import load_sklearn_model
                bundle = load_sklearn_model(model_path)
                self._model = bundle["model"]
                self._scaler = bundle["scaler"]

        def name(self) -> str:
            return "momentum_continuation"

        def alg_features(self) -> list[AlgorithmFeature]:
            return [
                AlgorithmFeature("alg_mc_signal", warmup=200,
                    description="Signed continuation signal [-1,1]"),
                AlgorithmFeature("alg_mc_probability", warmup=200,
                    description="Raw model probability [0,1]"),
                AlgorithmFeature("alg_mc_entropy_gate", warmup=200,
                    description="1 if entropy gate open, 0 if closed"),
            ]

        def required_columns(self) -> list[str]:
            return [
                "trend_momentum_300_last", "ent_tick_1m_mean",
                "whale_net_flow_4h_last", "trend_hurst_300_last",
                "trend_r_squared_300_last", "trend_monotonicity_300_last",
                "toxic_vpin_10_mean", "regime_accumulation_score_last",
                "ctx_funding_zscore_last",
            ]

        def step(self, tick: dict[str, float]) -> dict[str, float]:
            self._tick_count += 1

            ent = tick.get("ent_tick_1m_mean", float("nan"))
            gate_open = 1.0 if (not math.isnan(ent)
                                and ent < self._entropy_threshold) else 0.0

            if gate_open == 0.0 or self._model is None:
                return {
                    "alg_mc_signal": 0.0,
                    "alg_mc_probability": 0.5,
                    "alg_mc_entropy_gate": gate_open,
                }

            mom = tick.get("trend_momentum_300_last", 0.0)
            features = np.array([[
                mom * (1.0 - ent),                          # interaction
                tick.get("whale_net_flow_4h_last", 0.0)
                    * (1.0 if mom >= 0 else -1.0),          # whale confirm
                tick.get("trend_hurst_300_last", 0.5),
                math.sqrt(abs(mom)
                    * tick.get("trend_r_squared_300_last", 0.0)), # trend strength
                tick.get("trend_r_squared_300_last", 0.0),
                tick.get("trend_monotonicity_300_last", 0.5),
                tick.get("toxic_vpin_10_mean", 0.5),
                tick.get("regime_accumulation_score_last", 0.0),
                tick.get("ctx_funding_zscore_last", 0.0),
            ]])

            if self._scaler is not None:
                features = self._scaler.transform(features)

            prob = float(self._model.predict_proba(features)[0, 1])

            if prob > self._p_long:
                signal = (prob - 0.5) * 2.0
            elif prob < self._p_short:
                signal = -(0.5 - prob) * 2.0
            else:
                signal = 0.0

            return {
                "alg_mc_signal": signal,
                "alg_mc_probability": prob,
                "alg_mc_entropy_gate": gate_open,
            }

        def reset(self) -> None:
            self._tick_count = 0

================================================================================
1.8  RISK ANALYSIS
================================================================================

  Overfitting risk: MEDIUM
    9 features with logistic regression is moderate-dimensional. The entropy
    gate reduces the effective sample size (only low-entropy bars train the
    model), which increases overfitting risk.
    Mitigation: L2 regularization (C=0.1), walk-forward validation,
    OOS/IS ratio >= 0.7 hard gate.

  Data leakage risk: LOW
    Features are computed from current and past data only (no look-ahead in
    the Rust ingestor). Forward returns use future prices but are only used
    for labels, not features.
    Mitigation: embargo_bars=100 between train and validation windows.

  Regime mismatch risk: MEDIUM
    The model trains on low-entropy bars but the entropy gate is applied at
    inference time. If the entropy distribution shifts (e.g., Hyperliquid
    adds new market makers, reducing entropy overall), the gate threshold
    may become too restrictive or too permissive.
    Mitigation: monitor entropy distribution monthly, adjust threshold.
    Trigger: if fraction of bars passing entropy gate changes by > 20%
    vs training period -> alert and recalibrate threshold.

  Model decay risk: MEDIUM
    Momentum factor returns are cyclical. The model may learn patterns that
    are specific to the current market structure (e.g., specific whales,
    specific funding rate dynamics) and decay as participants change.
    Mitigation: weekly retraining, decay detection via rolling 7-day Sharpe.
    Trigger: if 7-day Sharpe < 0.0 for 3 consecutive windows -> disable.

  2. MEAN-REVERSION / FALSE-BREAKOUT DETECTOR

================================================================================

Status:       PROPOSED
Created:      2026-04-04 (design), 2026-06-03 (expanded)
File:         scripts/algorithms/mean_reversion_detector.py
Signal:       alg_mr_signal
Polarity:     high_long (positive signal = price will revert upward)
ML Method:    LightGBM (gradient boosted trees)
Gate:         Only active in high-entropy regimes (ent_tick_1m_mean > 0.70)

================================================================================
2.1  THESIS
================================================================================

In high-entropy regimes, price excursions from the moving average are noise,
not signal. Breakouts fail because there is no directional flow to sustain
them — the order book is balanced, market makers are active, and price
reverts to the mean.

The key insight is that not all high-entropy overextensions revert equally.
The reversion probability depends on a NON-LINEAR combination of:
  - How far price has deviated from the mean (z-score)
  - Whether liquidation clusters are positioned to amplify the move
  - Whether whales are diverging from the price move
  - Whether the move is crowded (high concentration)

A linear model misses the critical interactions: a z-score of 2.5 with low
liquidation risk is a genuine breakout starting, while a z-score of 2.5 with
high liquidation risk and whale divergence is a false breakout about to
snap back. LightGBM captures these interaction effects automatically via
tree splits.

Crypto-native angle:
  Liquidation cascades are unique to leveraged perpetual futures. When
  price overextends into a region with clustered liquidation stops, it
  can trigger a cascade that temporarily pushes price further — but once
  the liquidations clear, price snaps back violently. The combination of
  liquidation asymmetry + overextension is a crypto-native reversion signal.

================================================================================
2.2  SIGNAL CONSTRUCTION
================================================================================

2.2.1  Target variable
---------------------------------------------------------------------

  y_t = sign(P_{t+delta} - P_t)

  where delta = reversion horizon = 100 bars at 5min = 500min.

  For mean-reversion, we also define the reversion quality label:

    y_reversion = 1  if  sign(P_{t+delta} - P_t) != sign(P_t - MA_t)
                 0  otherwise

  This labels a bar as "reversion occurred" if the forward move is in the
  OPPOSITE direction of the current deviation from MA. This is a stricter
  label than just directional prediction — it specifically identifies
  reversion events.

2.2.2  Feature vector
---------------------------------------------------------------------

  Feature                             Code name                   Formula / Source
  -----------------------------------------------------------------------
  1. Mean-reversion z-score           zscore
     = (raw_midprice_last - mf_ema_15m_last) / vol_returns_5m_last

     Distance from 15m EMA in volatility-normalized units.

     Numerical examples:
       midprice = 67500, ema_15m = 67400, vol_5m = 0.0015 (0.15%)
       zscore = (67500 - 67400) / (67400 * 0.0015) = 100 / 101.1 = +0.99

       midprice = 67200, ema_15m = 67400, vol_5m = 0.0015
       zscore = (67200 - 67400) / (67400 * 0.0015) = -200 / 101.1 = -1.98

  2. Range position (breakout indicator) range_pos
     = regime_range_pos_24h_last

     Position of price within the 24h range: 0 = at bottom, 1 = at top.
     Values > 0.9 or < 0.1 = at range extremes = likely false breakout.

  3. Liquidation asymmetry            liq_asymmetry
     If heatmap features available:
       = hm_asymmetric_cascade_pot_last
     Else use regime features:
       = (liquidation_risk_above_2pct - liquidation_risk_below_2pct)
         / (liquidation_risk_above_2pct + liquidation_risk_below_2pct + 1e-10)

     Positive = more liquidations above price = upside risk.
     Negative = more liquidations below = downside risk.

  4. Concentration-momentum interaction concentration_momentum
     = conc_hhi_last * trend_momentum_300_last

     When HHI is high (concentrated market), momentum is driven by few
     participants — fragile, more likely to revert when they stop.

  5. Whale divergence                 whale_divergence
     = -sign(zscore) * whale_net_flow_4h_last

     Positive when whales are moving OPPOSITE to the deviation.
     If price is above MA (zscore > 0) and whales are selling
     (whale_flow < 0), this is positive — confirming reversion.

     Numerical example:
       zscore = +2.0 (overextended up), whale_flow = -1.5e6 (selling)
       whale_divergence = -sign(2.0) * (-1.5e6) = -1 * -1.5e6 = +1.5e6
       (positive = whales confirm reversion)

  6. Toxicity                         toxic_vpin_50_mean
     High VPIN during overextension suggests the move is driven by
     informed flow — less likely to revert (the informed trader knows
     something). Low VPIN = uninformed = more likely to revert.

  7. Distribution score               regime_distribution_score_last
     From Rust regime detection. High distribution score = whales are
     distributing (selling into strength) = reversion likely.

  8. Realized volatility              vol_returns_5m_mean
     Context feature. High vol = noisier environment = wider stop needed.
     Also used for position sizing (inverse proportional).

  Total: 8 features.

2.2.3  Model specification — LightGBM
---------------------------------------------------------------------

  Objective:     binary (log loss)
  Metric:        binary_logloss (primary), auc (secondary)

  Loss function:

    L = -(1/N) * sum_{i=1}^{N} [
        y_i * log(p_i) + (1 - y_i) * log(1 - p_i)
    ]

  where p_i = F(x_i) is the LightGBM ensemble prediction after sigmoid.

  Gradient (per sample, used by LightGBM internally):
    g_i = p_i - y_i
    h_i = p_i * (1 - p_i)     (Hessian for Newton step)

  Hyperparameters (defaults and sweep):

    Parameter           Default   Sweep range       Rationale
    -----------------------------------------------------------------------
    max_depth           5         [3, 5, 7]         Deeper = more interactions
    num_leaves          31        [15, 31, 63]      2^depth - 1 for balanced
    learning_rate       0.05      [0.01, 0.05, 0.1] Lower = more robust
    n_estimators        200       [100, 200, 400]   Early stopping overrides
    min_child_samples   50        [20, 50, 100]     Higher = less overfit
    subsample           0.8       [0.6, 0.8, 1.0]   Row sampling per tree
    colsample_bytree    0.8       [0.6, 0.8, 1.0]   Feature sampling per tree
    reg_alpha           0.1       [0.0, 0.1, 1.0]   L1 regularization
    reg_lambda          0.1       [0.0, 0.1, 1.0]   L2 regularization

  Early stopping: patience = 20 on validation loss, eval_set = last 2 days.

  Why LightGBM not logistic regression:
    The reversion signal depends on INTERACTION effects:
      - z-score alone does not predict reversion (could be a breakout)
      - z-score * liq_asymmetry = strong reversion predictor
      - z-score * whale_divergence * low_vpin = even stronger

    LightGBM learns these tree splits automatically. Logistic regression
    would need hand-engineered interaction features (z*liq, z*whale, etc.)
    and would still miss higher-order combinations.

  SHAP feature importance protocol:
    After training, compute SHAP values via shap.TreeExplainer.
    Expected ranking: zscore > liq_asymmetry > whale_divergence > vpin.
    If zscore is NOT in the top 3 features by mean |SHAP|, the model
    is learning noise — discard and retrain with fewer features.

2.2.4  Signal generation
---------------------------------------------------------------------

  P(reversion) = model.predict_proba(features)[0, 1]

  Combined signal:

    if |zscore| > 2.0 AND P(reversion) > 0.65 AND ent_tick_1m > 0.70:
        signal = -sign(zscore) * P(reversion)
        # negative sign: fade the deviation direction
    elif range_pos > 0.90 AND liq_asymmetry > 0.50 AND ent_tick_1m > 0.70:
        signal = -1.0 * P(reversion)
        # at top of range with liquidation risk above = short
    elif range_pos < 0.10 AND liq_asymmetry < -0.50 AND ent_tick_1m > 0.70:
        signal = +1.0 * P(reversion)
        # at bottom of range with liquidation risk below = long
    else:
        signal = 0.0

  Numerical example:
    zscore = +2.3, P(reversion) = 0.78, ent = 0.82
    signal = -sign(2.3) * 0.78 = -1.0 * 0.78 = -0.78  (short, fade the up-move)

    zscore = -1.5, P(reversion) = 0.60, ent = 0.75
    signal = 0.0  (|zscore| < 2.0, below threshold)

2.5  RISK MANAGEMENT
================================================================================

  Stop loss:
    stop_price = entry_price +/- (stop_loss_atr_mult * mf_atr_5m_last)

    For a short reversion trade (fading an up-move):
      stop = entry + 1.5 * ATR
    For a long reversion trade (fading a down-move):
      stop = entry - 1.5 * ATR

    ATR-based stops adapt to current volatility regime.
    Numerical example:
      entry = 67500 (short), ATR_5m = 80 bps of price = $54
      stop = 67500 + 1.5 * 54 = 67500 + 81 = $67,581

  Target:
    z-score = 0 (return to EMA). Close when:
      |raw_midprice - mf_ema_15m| / (vol * price) < 0.3

  Position size:
    size = base_size / vol_returns_5m_mean
    Higher volatility = smaller position (inverse vol sizing).

2.8  RISK ANALYSIS
================================================================================

  Overfitting risk: MEDIUM-HIGH
    LightGBM with 8 features and 200 trees can memorize patterns.
    Mitigation: min_child_samples=50, subsample=0.8, early stopping,
    regularization (reg_alpha, reg_lambda), OOS/IS ratio check.

  Regime mismatch risk: MEDIUM
    The model trains on high-entropy bars. If entropy threshold is wrong,
    it may include trending bars (where reversion fails).
    Mitigation: entropy gate with strict threshold (0.70).

  Liquidation data dependency risk: LOW
    If heatmap features are not available (optional category), the
    algorithm falls back to regime-based liquidation proxy.
    Mitigation: two code paths (heatmap vs regime fallback).

  Catastrophic loss risk: LOW
    Stop loss at 1.5x ATR limits single-trade loss. Position sizing
    by inverse volatility further limits exposure.

================================================================================
2.9  RESEARCH DIRECTIONS
================================================================================

  2.9.1  Adaptive z-score threshold
    Instead of fixed 2.0, use the rolling P95 of |zscore| over 7 days.
    Markets with persistently high z-scores (trending) need higher thresholds.

  2.9.2  Time-of-day conditioning
    Reversion may be stronger in specific sessions (e.g., US equity open
    causes crypto volatility that reverts after 2h). Add hour-of-day as feature.

  2.9.3  Multi-timeframe z-score
    Compute z-scores at 1m, 5m, 15m EMAs. When all three agree (overextended
    at all scales), reversion probability is higher.

  3. META-LABELING SYSTEM

================================================================================

Status:       PROPOSED
Created:      2026-04-04 (design), 2026-06-03 (expanded)
File:         scripts/algorithms/meta_labeler.py
Signal:       alg_meta_signal
Polarity:     inherits from base algorithm
ML Method:    Logistic Regression (interpretable)
Concept:      De Prado (2018) "Advances in Financial Machine Learning", Ch. 3

================================================================================
3.1  THESIS
================================================================================

Instead of predicting DIRECTION, predict WHETHER A SIMPLE SIGNAL WILL WORK.

Every existing algorithm (jump_detector, optimal_entry, funding_reversion)
fires many signals. Some lead to profit, some to loss. The question is not
"which direction?" but "is this a good time to trade at all?"

The meta-labeling system wraps any base algorithm with a second classifier
that learns to predict P(base signal leads to profit | current market state).
The base signal provides direction; the meta-label provides precision filtering.

This is the De Prado "triple barrier" framework adapted for our tick-level
algorithms. The key advantage: it can improve ANY existing algorithm without
modifying it. Wrapping jump_detector with a meta-label could filter out the
BTC losses (where jump_detector has only 54% win rate) while keeping the
ETH/SOL wins (where it has 62-69% win rate).

Why not just improve the base algorithm?
  The base algorithms are hypothesis-driven with interpretable signals.
  Making them more complex risks overfitting to noise. Meta-labeling keeps
  the base signal simple and adds a SEPARATE precision filter that learns
  from the market STATE (entropy, toxicity, concentration) — orthogonal
  information that the base algorithms don't consider.

================================================================================
3.2  SIGNAL CONSTRUCTION
================================================================================

3.2.1  Triple-barrier labeling (De Prado method)
---------------------------------------------------------------------

  For each tick t where the base algorithm fires (|base_signal| > 0):

  Define three barriers:

    Upper barrier:  P_t * (1 + profit_target)
    Lower barrier:  P_t * (1 - stop_loss)
    Time barrier:   t + max_holding_bars

  The label is determined by which barrier is hit first:

    y_meta = 1  if upper barrier hit first  (profit taken)
    y_meta = 0  if lower barrier hit first  (stopped out)
    y_meta = 0  if time barrier hit first AND return < 0  (time decay)
    y_meta = 1  if time barrier hit first AND return > 0  (slow profit)

  Default barrier parameters:
    profit_target:    5 bps (50 ticks at 100ms)
    stop_loss:        10 bps
    max_holding_bars: 100 (at 5min bars = 500min)

  The asymmetric barriers (profit < stop) reflect that we want to be
  selective: we'd rather miss a marginal trade than take a losing one.

  Numerical example:
    base_signal fires long at price $67,500.
    Upper barrier = 67500 * 1.0005 = $67,533.75
    Lower barrier = 67500 * 0.9990 = $67,432.50
    Time barrier  = 100 bars from now.

    Scenario A: price hits $67,534 after 20 bars -> y_meta = 1
    Scenario B: price hits $67,430 after 8 bars  -> y_meta = 0
    Scenario C: after 100 bars, price = $67,510  -> y_meta = 1 (small gain)
    Scenario D: after 100 bars, price = $67,480  -> y_meta = 0 (small loss)

3.2.2  Meta-label feature vector (market state)
---------------------------------------------------------------------

  These features describe the MARKET STATE at the time the base signal fires.
  They are orthogonal to the base signal's direction — they answer "is this
  a good environment for trading?"

  Feature                             Code name                   Purpose
  -----------------------------------------------------------------------
  1. Entropy (predictability)         ent_tick_1m_mean            Low = more predictable
  2. Entropy rate of change           ent_rate_of_change_5s_mean  Rising = regime shift
  3. Toxicity (informed trading)      toxic_vpin_10_mean          Low = less adverse selection
  4. Toxicity index                   toxic_index_mean            Composite toxicity measure
  5. Concentration (crowding)         conc_hhi_last               High = fragile move
  6. Whale agreement                  whale_directional_agreement_last
                                                                   High = smart money confirms
  7. Volatility                       vol_returns_5m_mean         Risk context
  8. Vol regime (short/long ratio)    vol_ratio_short_long_last   Expanding vs contracting
  9. Regime clarity                   regime_clarity_last         High = clear regime
  10. Spread (execution cost proxy)   raw_spread_bps_mean         Tight = cheap to trade

  Total: 10 state features (none are directional).

3.2.3  Model specification — Logistic Regression
---------------------------------------------------------------------

  Loss function:

    L(beta) = -(1/N) * sum_{i=1}^{N} [
        y_i * log(sigma(z_i)) + (1 - y_i) * log(1 - sigma(z_i))
    ] + (lambda/2) * ||beta||_2^2

  where:
    z_i = beta_0 + sum_{j=1}^{10} beta_j * x_{ij}
    sigma(z) = 1 / (1 + exp(-z))

  Regularization: C = 1.0 (lighter than momentum classifier because
  meta-labeling has more training samples — every base signal fire is
  a training example).

  Why logistic regression and NOT LightGBM:
    Interpretability is critical here. The meta-label tells us WHICH market
    state features predict signal failure. Logistic regression coefficients
    are directly interpretable:
      - beta_vpin > 0 means high toxicity predicts signal success (unusual)
      - beta_entropy < 0 means low entropy predicts success (expected)
    If the signs don't match intuition, the model is learning noise.

  Probability output:

    P(base signal success) = sigma(beta_0 + beta^T * state_features)

  Decision rule:

    If P(success) > meta_threshold:   pass through base signal
    If P(success) <= meta_threshold:  suppress base signal (output 0.0)

    Default meta_threshold = 0.55 (slightly above 0.5 to be conservative).

  Numerical examples:
    base_signal = +0.72 (jump_detector fires long)
    state: ent = 0.15, vpin = 0.30, hhi = 0.02, whale_agree = 0.8
    P(success) = 0.71
    -> 0.71 > 0.55 -> pass through: alg_meta_signal = +0.72

    base_signal = +0.65 (jump_detector fires long)
    state: ent = 0.80, vpin = 0.85, hhi = 0.15, whale_agree = 0.2
    P(success) = 0.38
    -> 0.38 < 0.55 -> suppress: alg_meta_signal = 0.0

3.2.4  Anti-leakage protocol
---------------------------------------------------------------------

  CRITICAL: the meta-label classifier must NOT see the same data used to
  evaluate the base signal. The training pipeline must use PURGED samples:

    1. Split data into K=5 folds
    2. For each fold i:
       a) Run base algorithm on ALL data (base algo has no ML, so no leakage)
       b) Generate triple-barrier labels on fold i
       c) Train meta-classifier on folds {1..K} \ {i}
       d) Generate meta-predictions on fold i
    3. Concatenate out-of-fold meta-predictions
    4. Evaluate precision lift vs unfiltered base signal

  This is the same anti-leakage protocol used in stacking (#9).

    from __future__ import annotations
    import math
    import numpy as np
    from .base import AlgorithmFeature, MicrostructureAlgorithm
    from .registry import get_algorithm

    class MetaLabeler(MicrostructureAlgorithm):
        """Wraps any base algorithm with a precision filter.

        The meta-label classifier learns P(base signal leads to profit)
        from market state features (entropy, toxicity, concentration).
        Passes through the base signal only when P(success) > threshold.

        References:
          De Prado, M.L. (2018) — Advances in Financial Machine Learning,
          Chapter 3: Meta-Labeling. Wiley.
        """

        def __init__(
            self,
            base_algorithm: str = "jump_detector",
            model_path: str | None = None,
            meta_threshold: float = 0.55,
            **base_kwargs,
        ):
            self._base = get_algorithm(base_algorithm, **base_kwargs)
            self._meta_model = None
            self._meta_scaler = None
            self._threshold = meta_threshold

            if model_path:
                from scripts.utils.model_io import load_sklearn_model
                bundle = load_sklearn_model(model_path)
                self._meta_model = bundle["model"]
                self._meta_scaler = bundle["scaler"]

        def name(self) -> str:
            return f"meta_labeler_{self._base.name()}"

        def alg_features(self) -> list[AlgorithmFeature]:
            return [
                AlgorithmFeature("alg_meta_signal", warmup=200,
                    description="Filtered base signal [-1,1]"),
                AlgorithmFeature("alg_meta_probability", warmup=200,
                    description="P(base signal success)"),
                AlgorithmFeature("alg_meta_base_raw", warmup=200,
                    description="Unfiltered base signal"),
                AlgorithmFeature("alg_meta_active", warmup=200,
                    description="1 if base signal fired"),
            ]

        def required_columns(self) -> list[str]:
            base_cols = self._base.required_columns()
            state_cols = [
                "ent_tick_1m_mean", "ent_rate_of_change_5s_mean",
                "toxic_vpin_10_mean", "toxic_index_mean",
                "conc_hhi_last", "whale_directional_agreement_last",
                "vol_returns_5m_mean", "vol_ratio_short_long_last",
                "regime_clarity_last", "raw_spread_bps_mean",
            ]
            return list(set(base_cols + state_cols))

        def step(self, tick: dict[str, float]) -> dict[str, float]:
            base_out = self._base.step(tick)
            base_primary = self._base.alg_features()[0].name
            base_signal = base_out.get(base_primary, 0.0)

            active = 1.0 if abs(base_signal) > 0.01 else 0.0

            if active == 0.0 or self._meta_model is None:
                return {
                    "alg_meta_signal": 0.0,
                    "alg_meta_probability": 0.5,
                    "alg_meta_base_raw": base_signal,
                    "alg_meta_active": active,
                }

            state = np.array([[
                tick.get("ent_tick_1m_mean", 0.5),
                tick.get("ent_rate_of_change_5s_mean", 0.0),
                tick.get("toxic_vpin_10_mean", 0.5),
                tick.get("toxic_index_mean", 0.5),
                tick.get("conc_hhi_last", 0.0),
                tick.get("whale_directional_agreement_last", 0.0),
                tick.get("vol_returns_5m_mean", 0.01),
                tick.get("vol_ratio_short_long_last", 1.0),
                tick.get("regime_clarity_last", 0.0),
                tick.get("raw_spread_bps_mean", 1.0),
            ]])

            if self._meta_scaler is not None:
                state = self._meta_scaler.transform(state)

            prob = float(self._meta_model.predict_proba(state)[0, 1])

            if prob > self._threshold:
                signal = base_signal
            else:
                signal = 0.0

            return {
                "alg_meta_signal": signal,
                "alg_meta_probability": prob,
                "alg_meta_base_raw": base_signal,
                "alg_meta_active": active,
            }

        def reset(self) -> None:
            self._base.reset()

  Note: MetaLabeler is NOT registered with @register because it wraps
  a dynamic base algorithm. Instead, instantiate directly:
    meta = MetaLabeler(base_algorithm="jump_detector",
                       model_path="models/meta_jump.pkl")

================================================================================

  Phase 3 — False positive reduction:

================================================================================

  Overfitting risk: LOW
    10 features, logistic regression, large sample size (every base signal
    fire is a training example). This is deliberately simple.

  Information leakage risk: MEDIUM
    The anti-leakage protocol (Section 3.2.4) is critical. If meta-label
    is trained on data that was also used to evaluate the base signal,
    the precision lift will be inflated in-sample and vanish OOS.
    Mitigation: strictly purged K-fold with embargo.

  Base signal degradation risk: LOW

    is wrong, the worst case is that it suppresses a good signal
    (opportunity cost, not loss). It never creates new signals.

  Over-suppression risk: MEDIUM
    If the meta-threshold is too high, the meta-label suppresses too many
    trades, leaving insufficient sample size for the base algorithm to
    demonstrate its edge.
    Mitigation: monitor trade count. If meta fires < 20% of base, lower
    threshold.

  4. REGIME STATE MACHINE

================================================================================

Status:       PROPOSED
Created:      2026-04-04 (design), 2026-06-03 (expanded)
File:         scripts/algorithms/regime_state_machine.py
Signal:       alg_rsm_regime (discrete state) + alg_rsm_confidence
Polarity:     N/A (gating mechanism, not directional signal)
ML Method:    Hidden Markov Model (HMM) or manual feature thresholds
States:       6 discrete market regimes

================================================================================
4.1  THESIS
================================================================================

Markets transition between discrete regimes. A single model that averages
over all regimes produces a weak compromise signal — like wearing a winter
coat averaged with a swimsuit. Within a specific regime, the appropriate
strategy is clear; the hard part is identifying WHICH regime you're in.

This algorithm is a GATING mechanism, not a signal generator. It answers
"which strategy should be active?" not "what should I trade?" The downstream
algorithms (#1 Momentum Continuation and #2 Mean-Reversion Detector) do the
actual trading, conditioned on the regime.

Existing infrastructure:
  NAT already has regime detection at two levels:
    1. Rust ingestor: 20 regime_* features (absorption, divergence, churn,
       range position) computed at 1h/4h/24h windows.
    2. GMM classifier: 5-component GaussianMixture trained on
       [illiq_kyle_lambda_300, toxic_vpin_50, regime_absorption_zscore,
       trend_hurst_300, whale_net_flow_4h] — outputs 8 features including
       regime label and per-component probabilities.

  This algorithm COMPOSES these existing features into a higher-level
  6-state machine with explicit trading rules per state.

================================================================================
4.2  STATE SPACE DESIGN
================================================================================

  6 regimes with feature-based classification:

  State 0: ACCUMULATION
    Definition: Large volume absorbed without price rise. Whales buying
    at support levels. Low entropy (orderly), high absorption.
    Features:
      regime_accumulation_score > 0.7
      whale_net_flow_24h > 0
      regime_absorption_zscore > 1.0
      regime_range_pos_24h < 0.30
    Trading rule: LONG bias, wait for entropy drop to confirm.

  State 1: DISTRIBUTION
    Definition: Large volume absorbed without price fall. Whales selling
    into strength. Low entropy, high absorption, price at top of range.
    Features:
      regime_distribution_score > 0.7
      whale_net_flow_24h < 0
      regime_absorption_zscore > 1.0
      regime_range_pos_24h > 0.70
    Trading rule: SHORT bias, wait for entropy drop to confirm.

  State 2: TRENDING_UP
    Definition: Persistent upward momentum with whale alignment.
    Features:
      trend_momentum_300 > 0
      trend_hurst_300 > 0.55 (persistent)
      ent_tick_1m < 0.30
      whale_net_flow_4h > 0
    Trading rule: Momentum continuation (use Algorithm #1).

  State 3: TRENDING_DOWN
    Definition: Persistent downward momentum with whale alignment.
    Features:
      trend_momentum_300 < 0
      trend_hurst_300 > 0.55
      ent_tick_1m < 0.30
      whale_net_flow_4h < 0
    Trading rule: Short momentum (use Algorithm #1, short side).

  State 4: RANGING
    Definition: Two-sided flow, no direction. High entropy, high churn.
    Features:
      ent_tick_1m > 0.60
      trend_hurst_300 < 0.45 (anti-persistent)
      regime_churn_4h_zscore > 1.5
    Trading rule: Mean reversion (use Algorithm #2).

  State 5: VOLATILE_NOISE
    Definition: High volatility, high toxicity, uninformative.
    Features:
      vol_returns_5m > 2 * median(vol_returns_5m over 24h)
      toxic_vpin_50 > 0.80
      ent_tick_1m > 0.70
    Trading rule: NO TRADE. Wait for regime transition.

================================================================================
4.3  CLASSIFICATION METHOD — OPTION A: MANUAL THRESHOLDS
================================================================================

  Scoring function for each state s:

    score_s(t) = sum_{f in features_s} w_f * indicator(f(t) meets threshold_f)

  where w_f is the weight (default 1.0 for each condition).

  State assignment:

    state(t) = argmax_s { score_s(t) }

  Confidence:

    confidence(t) = max_score / (sum_of_all_scores + epsilon)

  where epsilon = 1e-10 to avoid division by zero.

  Numerical example:
    tick features: accumulation_score=0.8, whale_24h=+5e6,
                   absorption_z=1.5, range_pos=0.20

    score_ACCUMULATION = 1 + 1 + 1 + 1 = 4  (all conditions met)
    score_DISTRIBUTION = 0 + 0 + 1 + 0 = 1  (only absorption met)
    score_TRENDING_UP  = 0                    (entropy condition not met)
    ...

    state = ACCUMULATION, confidence = 4 / (4+1+0+0+0+0) = 0.80

  Tie-breaking: prefer VOLATILE_NOISE over other states (conservative —
  when ambiguous, don't trade).

================================================================================
4.4  CLASSIFICATION METHOD — OPTION B: HIDDEN MARKOV MODEL
================================================================================

4.4.1  Model specification
---------------------------------------------------------------------

  Hidden states:     Z_t in {0, 1, 2, 3, 4, 5}  (6 states)
  Observations:      X_t in R^6 (6-dimensional feature vector)
  Time index:        t = 1, 2, ..., T (bar index, 5min bars)

  Observation vector:
    X_t = [ent_tick_1m, trend_momentum_300, whale_net_flow_4h,
           regime_absorption_zscore, trend_hurst_300, vol_returns_5m]

  Model parameters theta = {A, mu, Sigma, pi}:

    Transition matrix:  A[i,j] = P(Z_t = j | Z_{t-1} = i)
      6x6 matrix. Each row sums to 1.
      Expected structure: high diagonal (regimes are sticky),
      off-diagonal captures regime transitions.

    Emission parameters (per state k):
      mu_k in R^6         (mean observation vector in state k)
      Sigma_k in R^{6x6}  (covariance of observations in state k)

    Initial distribution:  pi_k = P(Z_1 = k)

4.4.2  Baum-Welch training (EM algorithm)
---------------------------------------------------------------------

  Initialization:
    Use existing GMM labels from train_regime_gmm.py as initial state
    assignments. This gives a warm start and ensures the HMM states
    correspond to economically meaningful regimes.

  E-step (forward-backward):

    Forward variable:
      alpha_1(k) = pi_k * N(X_1; mu_k, Sigma_k)
      alpha_t(k) = [sum_j alpha_{t-1}(j) * A[j,k]] * N(X_t; mu_k, Sigma_k)

    Backward variable:
      beta_T(k) = 1
      beta_t(k) = sum_j A[k,j] * N(X_{t+1}; mu_j, Sigma_j) * beta_{t+1}(j)

    State posterior:
      gamma_t(k) = alpha_t(k) * beta_t(k) / P(X_{1:T})

    Transition posterior:
      xi_t(i,j) = alpha_t(i) * A[i,j] * N(X_{t+1}; mu_j, Sigma_j)
                  * beta_{t+1}(j) / P(X_{1:T})

  M-step (parameter update):

    A[i,j] = sum_{t=1}^{T-1} xi_t(i,j) / sum_{t=1}^{T-1} gamma_t(i)

    mu_k = sum_{t=1}^{T} gamma_t(k) * X_t / sum_{t=1}^{T} gamma_t(k)

    Sigma_k = sum_{t=1}^{T} gamma_t(k) * (X_t - mu_k)(X_t - mu_k)^T
              / sum_{t=1}^{T} gamma_t(k)

    pi_k = gamma_1(k)

  Convergence: iterate E-step and M-step until
    |log P(X_{1:T} | theta_new) - log P(X_{1:T} | theta_old)| < 1e-4
    or max 100 iterations.

4.4.3  Online inference (forward algorithm)
---------------------------------------------------------------------

  At each new tick t, compute the filtered state distribution:

    alpha_t(k) = [sum_j alpha_{t-1}(j) * A[j,k]] * N(X_t; mu_k, Sigma_k)

  Normalize: alpha_t(k) <- alpha_t(k) / sum_k alpha_t(k)

  State estimate:  state(t) = argmax_k alpha_t(k)
  Confidence:      confidence(t) = max_k alpha_t(k)

  Transition probability (next-step prediction):
    P(Z_{t+1} = j | X_{1:t}) = sum_k alpha_t(k) * A[k,j]

  This is O(K^2) per tick where K=6 — trivially fast.

4.4.4  Model selection (number of states)
---------------------------------------------------------------------

  Fit HMM with K = 3, 4, 5, 6, 7 states.
  Select K that minimizes BIC:

    BIC = -2 * log L + n_params * log(T)

  where:
    log L = log P(X_{1:T} | theta)   (log-likelihood from forward algorithm)
    n_params = K*K + K*6 + K*6*7/2   (transition + means + covariances)
             = K^2 + 6K + 21K        (for diagonal covariance)
    T = number of bars

  Expected: K = 4 or 5 (accumulation/distribution may merge with trending).

  Implementation: hmmlearn.hmm.GaussianHMM (already used in
  scripts/cluster_pipeline/transitions.py).

  Overfitting risk (HMM): MEDIUM
    HMM with 6 states and diagonal covariance has 6*6 + 6*6 + 6*6 = 108

    4032 samples, this is 37 samples per parameter — marginal.
    Mitigation: use diagonal covariance (fewer params), prefer K=4 if
    BIC is close. Warm-start from GMM labels.

  Overfitting risk (manual): LOW
    No parameters fitted to data. Thresholds are domain-knowledge based.
    Risk is that thresholds are wrong, not that they overfit.

  Regime flickering risk: MEDIUM
    If features are noisy near thresholds, the classifier may switch
    states rapidly (ACCUMULATION -> TRENDING -> ACCUMULATION within
    3 bars). This causes downstream algorithms to whipsaw.
    Mitigation: require state to persist for min_duration=5 bars before
    changing. Hysteresis: once in a state, require larger deviation to exit.

  Alignment risk: LOW
    The 6 states map to well-understood market concepts from microstructure
    literature (Wyckoff accumulation/distribution, trending, ranging).

  5. ONLINE ANOMALY / CHANGE-POINT DETECTOR

================================================================================

Status:       PROPOSED
Created:      2026-04-04 (design), 2026-06-03 (expanded)
File:         scripts/algorithms/changepoint_detector.py
Signal:       alg_cp_direction
Polarity:     high_long
ML Method:    CUSUM (Option A) / Bayesian Online Change-Point (Option B)
Horizon:      Real-time (tick-level detection)

================================================================================
5.1  THESIS
================================================================================

Regime transitions are the highest-value trading events. The Regime State
Machine (#4) classifies the CURRENT regime, but by the time it confidently
identifies a new regime, the transition is already well underway and the
best entry point has passed.

This algorithm detects the TRANSITION ITSELF — the moment when the market
shifts from one regime to another — using sequential change-point detection.
It is an early warning system: "something is changing" fires before "we are
now in a new regime."

The two most valuable transitions to detect:
  1. RANGING -> TRENDING:  entropy drops, momentum emerges. Enter early
     in the direction of the emerging trend.
  2. ACCUMULATION -> TRENDING_UP: whale buying completes, price breaks out.
     The breakout is real (unlike false breakouts in RANGING).

================================================================================
5.2  OPTION A: CUSUM (Cumulative Sum Control Chart)
================================================================================

5.2.1  Algorithm
---------------------------------------------------------------------

  For a monitored signal x_t with known in-control mean mu:

  Upper CUSUM (detects upward shift):
    S_t^+ = max(0, S_{t-1}^+ + (x_t - mu) - k)

  Lower CUSUM (detects downward shift):
    S_t^- = min(0, S_{t-1}^- + (x_t - mu) + k)

  Alarms:
    If S_t^+ > h  ->  positive shift detected (reset S_t^+ = 0)
    If S_t^- < -h ->  negative shift detected (reset S_t^- = 0)

  Parameters:
    mu = in-control mean (rolling mean over calibration_window)
    k  = allowance (slack): filters small fluctuations
    h  = decision threshold: higher = fewer false alarms, slower detection

  Relationship between k, h, and performance:
    ARL_0 = Average Run Length under H0 (no change)  = expected ticks between
            false alarms
    ARL_1 = Average Run Length under H1 (change of size delta) = expected
            ticks to detect a real change

    For detecting a shift of delta in the mean:
      Optimal k = delta / 2     (halfway between old and new mean)
      h controls the ARL_0/ARL_1 tradeoff:
        h = 4*sigma: ARL_0 ~ 1000, ARL_1 ~ 10-20 for delta = 2*sigma
        h = 5*sigma: ARL_0 ~ 5000, ARL_1 ~ 15-30

  Numerical example (entropy monitoring):
    mu = 0.65 (average ent_tick_1m over last 1000 ticks)
    sigma = 0.12
    k = 0.06 (= sigma/2)
    h = 0.48 (= 4*sigma)

    Tick 1: ent = 0.60, S^+ = max(0, 0 + (0.60 - 0.65) - 0.06) = max(0, -0.11) = 0
    Tick 2: ent = 0.55, S^+ = max(0, 0 + (0.55 - 0.65) - 0.06) = 0
    Tick 3: ent = 0.40, S^+ still 0 (entropy dropping, but S^+ detects UPWARD shift)

    For downward shift (entropy drop = trending regime starting):
    Tick 1: S^- = min(0, 0 + (0.60 - 0.65) + 0.06) = min(0, +0.01) = 0
    Tick 2: S^- = min(0, 0 + (0.55 - 0.65) + 0.06) = min(0, -0.04) = -0.04
    Tick 3: S^- = min(0, -0.04 + (0.40 - 0.65) + 0.06) = min(0, -0.23) = -0.23
    Tick 4: S^- = min(0, -0.23 + (0.35 - 0.65) + 0.06) = min(0, -0.47) = -0.47
    Tick 5: S^- = min(0, -0.47 + (0.30 - 0.65) + 0.06) = min(0, -0.76) = -0.76
    -> |S^-| = 0.76 > h = 0.48 -> ALARM: entropy dropped! Trending regime starting.

5.2.2  Monitored signals (3 parallel CUSUM instances)
---------------------------------------------------------------------

  Stream 1: Entropy (ent_tick_1m)
    Detects: entropy DROP = trending regime starting
    Action: prepare for directional trade
    mu = rolling mean over 1000 ticks, k = sigma/2, h = 4*sigma

  Stream 2: Whale flow (whale_net_flow_1h)
    Detects: whale flow SURGE = accumulation/distribution phase
    Action: align with whale direction
    mu = 0 (neutral), k = sigma/2, h = 4*sigma

  Stream 3: Absorption (regime_absorption_zscore)
    Detects: absorption SPIKE = large volume being absorbed
    Action: anticipate breakout direction from whale flow sign
    mu = 0 (z-score already centered), k = 0.5, h = 2.0

5.2.3  Multi-stream aggregation
---------------------------------------------------------------------

  Combined direction signal:

    if entropy_alarm AND whale_alarm:
        direction = sign(whale_net_flow_1h at alarm time)
        confidence = 1.0
    elif entropy_alarm:
        direction = sign(trend_momentum_300 at alarm time)
        confidence = 0.6
    elif whale_alarm AND absorption_alarm:
        direction = sign(whale_net_flow_1h)
        confidence = 0.7
    else:
        direction = 0
        confidence = 0.0

  Signal = direction * confidence

================================================================================
5.3  OPTION B: BAYESIAN ONLINE CHANGE-POINT DETECTION
================================================================================

5.3.1  Algorithm (Adams & MacKay 2007)
---------------------------------------------------------------------

  Maintain a distribution over "run length" r_t — the number of ticks
  since the last change-point.

  At each tick t:

  Step 1 — Predictive probability:
    For each possible run length r:
      pi_t(r) = P(x_t | x_{t-r:t-1})

    Using Student-t predictive distribution (conjugate to
    Normal-Inverse-Gamma prior):

      pi_t(r) = t_{2*alpha_r}(x_t; mu_r, sigma_r^2 * (kappa_r + 1) / kappa_r)

    where (mu_r, kappa_r, alpha_r, beta_r) are the sufficient statistics
    of the Normal-Inverse-Gamma posterior for run length r.

  Step 2 — Growth probability (run length increases by 1):
    P(r_t = r+1, x_{1:t}) = P(r_{t-1} = r, x_{1:t-1}) * (1 - H) * pi_t(r)

    H = hazard rate = 1/lambda (probability of change-point at each tick)
    lambda = expected run length = 500 (default, ~50 seconds at 100ms)

  Step 3 — Change-point probability (run length resets to 0):
    P(r_t = 0, x_{1:t}) = sum_r P(r_{t-1} = r, x_{1:t-1}) * H * pi_prior

    pi_prior = predictive probability under the prior (no data for new run)

  Step 4 — Normalize:
    P(r_t | x_{1:t}) = P(r_t, x_{1:t}) / sum_r P(r_t = r, x_{1:t})

  Step 5 — Change-point posterior:
    P(change at t) = P(r_t = 0 | x_{1:t})

    If P(change) > change_threshold (default 0.30) -> ALARM

  Sufficient statistics update (Normal-Inverse-Gamma):
    When run length increases (no change):
      mu_r <- (kappa_r * mu_r + x_t) / (kappa_r + 1)
      kappa_r <- kappa_r + 1
      alpha_r <- alpha_r + 0.5
      beta_r <- beta_r + kappa_r * (x_t - mu_r)^2 / (2 * (kappa_r + 1))

    When run length resets (change-point):
      Reset to prior: mu_0, kappa_0, alpha_0, beta_0

5.3.2  Computational complexity
---------------------------------------------------------------------

  Naive: O(t) per tick (run length can be 0 to t).
  Practical: truncate run length distribution at r_max = 500 ticks.
  Per-tick cost: O(r_max) = O(500) = constant.

  Memory: O(r_max * d) where d = dimensionality of sufficient statistics.
  For univariate monitoring: 4 floats per run length = 2000 floats = 16KB.

5.7  RISK ANALYSIS
================================================================================

  False alarm risk: MEDIUM
    CUSUM with h = 4*sigma has ARL_0 ~ 1000 ticks = ~100 seconds.
    At 3 symbols, this means ~1 false alarm per 33 seconds across the
    portfolio. Acceptable if downstream gating adds a confirmation
    requirement (e.g., require 2 of 3 streams to alarm).
    Mitigation: increase h to 5*sigma (ARL_0 ~ 5000).

  Missed detection risk: LOW (CUSUM) / MEDIUM (Bayesian)
    CUSUM is designed for mean-shift detection — its strength. It will
    miss variance-only changes (e.g., vol spike without mean change).
    Bayesian method detects both mean and variance changes.

  Computational risk: LOW (CUSUM) / MEDIUM (Bayesian)
    CUSUM is O(1) per tick per stream. Trivial.
    Bayesian is O(r_max) per tick. With r_max=500, ~2000 FLOPs/tick.
    At 30 ticks/sec * 3 symbols = 90 calls/sec * 2000 = 180K FLOPs/sec.
    Negligible on any modern CPU.

  6. NEAREST-NEIGHBOR STATE RETRIEVAL

================================================================================

Status:       PROPOSED
Created:      2026-04-04 (design), 2026-06-03 (expanded)
File:         scripts/algorithms/knn_retrieval.py
Signal:       alg_knn_signal
Polarity:     high_long
ML Method:    K-Nearest Neighbors with Mahalanobis distance
Nature:       Non-parametric, instance-based learning

================================================================================
6.1  THESIS
================================================================================

The current market state has historical analogs. If past periods that
"looked like" today (in feature space) tended to produce positive returns,
then today is likely to produce positive returns as well.

This is the simplest possible ML algorithm: no parameters are fitted, no
gradients are computed, no model is trained. The entire "model" is the
historical database of (state_vector, forward_return) pairs. At inference
time, find the K most similar historical states and aggregate their outcomes.

The algorithm is adaptive by construction: as new data arrives, the
neighbor pool updates automatically. There is no concept of "model decay"
because there is no model — only data.

Existing infrastructure:
  sklearn.neighbors.NearestNeighbors is already imported in the codebase.
  sklearn.covariance.LedoitWolf is available for robust covariance estimation
  (needed for Mahalanobis distance).

================================================================================
6.2  SIGNAL CONSTRUCTION
================================================================================

6.2.1  State vector (7 dimensions)
---------------------------------------------------------------------

  Dimension   Feature                        Range         Role
  -----------------------------------------------------------------------
  1           ent_tick_1m_mean               [0, ln(3)]    Regime
  2           trend_momentum_300_last        (-inf, inf)   Direction
  3           whale_net_flow_4h_last         (-inf, inf)   Smart money
  4           toxic_vpin_50_mean             [0, 1]        Toxicity
  5           trend_hurst_300_last           [0, 1]        Persistence
  6           regime_accumulation_score_last [0, 1]        Regime phase
  7           vol_returns_5m_mean            [0, inf)      Risk

  All features are bar-aggregated at 5min via aggregate_bars().

6.2.2  Distance metric — Mahalanobis
---------------------------------------------------------------------

  d(x, y) = sqrt((x - y)^T * S^{-1} * (x - y))

  where S is the covariance matrix of state vectors estimated from
  the historical database.

  Covariance estimation: Ledoit-Wolf shrinkage estimator
  (sklearn.covariance.LedoitWolf). This produces a well-conditioned
  covariance matrix even when features are correlated.

  Why Mahalanobis and not Euclidean:
    Feature scales differ by orders of magnitude:
      ent_tick_1m ~ 0.5 (range [0, 1.1])
      whale_net_flow_4h ~ 1e6 (range [-1e7, 1e7])
      vol_returns_5m ~ 0.001 (range [0, 0.05])

    Euclidean distance would be dominated by whale_net_flow_4h.
    Mahalanobis normalizes by the covariance structure, making each
    feature contribute proportionally to its information content.

  Numerical example:
    Current state x = [0.25, 0.01, 2e6, 0.3, 0.6, 0.8, 0.001]
    Neighbor  y1 = [0.22, 0.012, 1.8e6, 0.28, 0.58, 0.75, 0.0012]
    Neighbor  y2 = [0.70, -0.005, -1e6, 0.75, 0.45, 0.2, 0.003]

    d(x, y1) = small (similar regime, similar direction)
    d(x, y2) = large (different regime, opposite direction, different vol)

6.2.3  Neighbor outcome aggregation
---------------------------------------------------------------------

  For the K nearest neighbors {y_1, ..., y_K} with forward returns
  {r_1, ..., r_K}:

    expected_return = (1/K) * sum_{i=1}^{K} r_i

    win_rate = (1/K) * sum_{i=1}^{K} 1{r_i > 0}

    return_std = std({r_1, ..., r_K})

    confidence = |expected_return| / (return_std + 1e-10)

  Signal:
    if |expected_return| > cost_threshold AND win_rate > 0.60:
        signal = sign(expected_return) * min(confidence, 1.0)
    else:
        signal = 0.0

  Default: K = 20, cost_threshold = 2 bps.

  Numerical example:
    K = 20 nearest neighbors found.
    Returns: [+5, +3, -2, +8, +1, -4, +6, +2, -1, +7,
              +4, -3, +9, +2, +5, -2, +3, +6, +1, +4] (in bps)
    expected_return = +2.7 bps
    win_rate = 15/20 = 0.75
    return_std = 3.5 bps
    confidence = 2.7 / 3.5 = 0.77
    signal = +1 * 0.77 = +0.77  (long with moderate confidence)

6.2.4  Time-decay weighting (optional)
---------------------------------------------------------------------

  Older neighbors may be less relevant due to market structure evolution.
  Apply exponential decay:

    w_i = exp(-lambda * age_i)

  where age_i is the number of days since neighbor i was observed,
  lambda = decay rate (default 0.01 = half-life ~ 69 days).

  Weighted aggregation:
    expected_return = sum(w_i * r_i) / sum(w_i)
    win_rate = sum(w_i * 1{r_i > 0}) / sum(w_i)

    changes, the database adapts within 30 days.

  7. REGIME-CONDITIONED LIGHTGBM

================================================================================

Status:       PROPOSED
Created:      2026-04-04 (design), 2026-06-03 (expanded)
File:         scripts/algorithms/regime_conditioned_lgbm.py
Signal:       alg_rlgbm_signal
Polarity:     high_long
ML Method:    Separate LightGBM model per macro regime
Horizon:      5min to 1h
Data Needed:  14+ days of regime-profiled data

================================================================================
7.1  THESIS
================================================================================

Phase 1 (direct LightGBM on raw features) proved signal exists: +4.18% edge
over base rate, statistically significant. But it was unprofitable after costs
(-0.45 bps/trade). The problem: the global model averages over distinct
market regimes, producing a weak compromise prediction.

Within a TRENDING regime, momentum features are highly predictive. Within a
RANGING regime, mean-reversion features dominate. A global model uses both
feature sets and produces a signal that is neither good momentum nor good
reversion — just weak.

The solution: train SEPARATE models per regime, each seeing only data from
its own regime and using only the relevant features. If global edge is 4%,
within-regime edge may be 8-15%, which clears the cost threshold.

Connection to existing infrastructure:
  The GMM classifier (train_regime_gmm.py) already produces 5-component
  regime labels: Accumulation, Markup, Distribution, Markdown, Ranging.
  These labels are available in real-time via the Rust inference engine
  (rust/ing/src/ml/regime.rs). This algorithm consumes those labels to
  route to the appropriate per-regime model.

================================================================================
7.2  ARCHITECTURE
================================================================================

  Step 1 — Regime detection (existing):
    Read GMM label from regime feature (already computed):
      regime = int(tick["regime"])
      regime_confidence = tick["regime_confidence"]

    Regime map:
      0 = Accumulation, 1 = Markup, 2 = Distribution,
      3 = Markdown, 4 = Ranging

  Step 2 — Model routing:
    If regime_confidence > confidence_threshold (default 0.6):
      Use model_{regime} to predict forward return.
    Else:
      Use model_global as fallback (trained on all regimes).

  Step 3 — Per-regime feature selection:

    Regime 0+1 (Accumulation/Markup — bullish):
      Features: trend_momentum_300, trend_hurst_300, whale_net_flow_4h,
                regime_accumulation_score, trend_r_squared_300,
                mf_rsi_5m, ctx_funding_zscore
      Rationale: momentum and whale features predict continuation

    Regime 2+3 (Distribution/Markdown — bearish):
      Features: trend_momentum_300, trend_hurst_300, whale_net_flow_4h,
                regime_distribution_score, toxic_vpin_50,
                mf_rsi_5m, ctx_funding_zscore
      Rationale: distribution score and toxicity predict reversal/continuation

    Regime 4 (Ranging):
      Features: vol_returns_5m, toxic_vpin_50, regime_churn_4h_zscore,
                mf_bb_pctb_5m, imbalance_qty_l1, conc_hhi,
                ent_tick_1m
      Rationale: mean-reversion features (Bollinger, imbalance, vol)

    Global fallback:
      Features: all features from all regimes (union)
      Used when regime_confidence < threshold

  Step 4 — Signal generation:
    predicted_return = model_{regime}.predict(features)
    signal = clip(predicted_return / cost_threshold, -1, 1)

================================================================================
7.3  TRAINING PROTOCOL
================================================================================

  Per-regime training:

    For regime R in {0, 1, 2, 3, 4}:
      data_R = bars where GMM regime == R AND regime_confidence > 0.6
      if len(data_R) < min_samples (default 500):
          merge R into closest regime (0+1, 2+3, or skip)
      model_R = LightGBM.fit(data_R[features_R], data_R[forward_return])

  Handling regime imbalance:
    Expected distribution: ~20% each regime (uniform GMM with 5 components).
    If any regime has < 10% of data, merge with its neighbor:
      Accumulation + Markup -> "Bullish"
      Distribution + Markdown -> "Bearish"
      Ranging stays solo

    With 14 days * 288 bars = 4032 total bars, 10% = 403 bars.
    LightGBM with 7 features and 200 trees needs ~500+ bars to avoid
    overfitting. This is marginal — use stronger regularization for
    small-sample regimes (min_child_samples=100 instead of 50).

  Walk-forward:
    Train: 14 days of regime-profiled bars
    Validation: 3 days
    Embargo: 100 bars
    Retrain: weekly
    Per-fold: retrain all per-regime models and the global fallback

    Multiple models (5 per-regime + 1 global) with limited data per regime.
    This is the highest-risk algorithm in the catalogue.
    Mitigation: aggressive regularization, minimum sample requirements,
    merge rare regimes, compare per-regime vs global on OOS data.

  Regime label noise risk: MEDIUM
    GMM labels are probabilistic. Near regime boundaries, the label may
    flicker between regimes, causing the wrong model to be used.
    Mitigation: confidence threshold (default 0.6), fallback to global.

  Complexity risk: MEDIUM
    6 models to train, maintain, and monitor. Any one of them could
    degrade without being noticed.
    Mitigation: per-model monitoring (rolling 7-day Sharpe per regime model).

  8. HMM WITH EMISSIONS

================================================================================

Status:       PROPOSED
Created:      2026-04-04 (design), 2026-06-03 (expanded)
File:         scripts/algorithms/hmm_emissions.py
Signal:       alg_hmm_signal
Polarity:     high_long
ML Method:    Hidden Markov Model with Gaussian emissions
Horizon:      30min to 4h (bar-level)
Data Needed:  7+ days

================================================================================
8.1  THESIS
================================================================================

Unlike Algorithm #4 (Regime State Machine) which classifies regimes from
observable features and then applies rules, and unlike Algorithm #7 which
uses regime labels to route to per-regime models, this HMM learns regimes,
their transition probabilities, AND their return distributions JOINTLY.

The key advantage: the HMM defines regimes based on what produces the
best GENERATIVE model of observed data — not based on human intuition about
what a "regime" should look like. If the data says there are 3 regimes (not
6), the HMM will learn 3 regimes. If one of those regimes has strongly
positive returns, that's the regime to be long in.

The transition matrix provides PREDICTIVE information: if regime A typically
transitions to regime B, and regime B has negative returns, then being in
regime A with increasing transition probability to B is a SHORT signal —
even though regime A itself may be neutral.

Existing infrastructure:
  hmmlearn.hmm.GaussianHMM is already used in
  scripts/cluster_pipeline/transitions.py for transition modeling.

================================================================================
8.2  MODEL SPECIFICATION
================================================================================

  Hidden states:   Z_t in {1, 2, ..., K}   (K selected by BIC, expect 3-5)
  Observations:    X_t in R^d              (d = 7, see below)
  Time index:      t = 1, 2, ..., T        (5min bars)

  Observation vector:
    X_t = [forward_return_5min,
           ent_tick_1m_mean,
           trend_momentum_300_last,
           vol_returns_5m_mean,
           whale_net_flow_4h_last,
           imbalance_qty_l1_mean,
           toxic_vpin_10_mean]

  Keep dimensionality at d=7. Higher dimensions require more data for
  reliable covariance estimation. If more features are needed, use PCA
  to compress to d=7 first.

  Model parameters theta = {A, mu, Sigma, pi}:

    A in R^{K x K}:     transition matrix, A[i,j] = P(Z_t = j | Z_{t-1} = i)
    mu_k in R^d:         emission mean for state k
    Sigma_k in R^{d x d}: emission covariance for state k
    pi in R^K:           initial state distribution

  Total parameters (diagonal covariance):
    K^2 + K*d + K*d + K = K^2 + 2Kd + K
    For K=4, d=7: 16 + 56 + 56 + 4 = 132 parameters
    With 7 days * 288 bars = 2016 samples -> 15 samples per parameter (tight)
    With 14 days -> 30 samples per parameter (adequate)

================================================================================
8.3  TRAINING — BAUM-WELCH (EM)
================================================================================

  See Algorithm #4, Section 4.4.2 for the full E-step/M-step equations.
  The procedure is identical.

  Initialization:
    Warm-start from existing GMM labels (train_regime_gmm.py):
      For each GMM component k:
        mu_k = mean of observations in component k
        Sigma_k = covariance of observations in component k
      A initialized as uniform with diagonal bias:
        A[i,j] = 0.8 if i==j, else 0.2/(K-1)
      pi = uniform

  Convergence: |delta log-likelihood| < 1e-4 or max 100 iterations.

  Implementation:
    from hmmlearn.hmm import GaussianHMM

    model = GaussianHMM(
        n_components=K,
        covariance_type="diag",   # start with diagonal
        n_iter=100,
        tol=1e-4,
        init_params="",           # we provide our own initialization
    )
    model.means_ = initial_means
    model.covars_ = initial_covars
    model.transmat_ = initial_A
    model.startprob_ = initial_pi
    model.fit(X_train)

================================================================================
8.4  INFERENCE AND SIGNAL GENERATION
================================================================================

8.4.1  Online state estimation (forward algorithm)
---------------------------------------------------------------------

  At each bar t, compute:
    alpha_t(k) = P(Z_t = k | X_{1:t})

  This is the filtered state distribution. State estimate:
    state(t) = argmax_k alpha_t(k)

8.4.2  Expected return signal
---------------------------------------------------------------------

  Method A — Emission-based:
    E[return | X_{1:t}] = sum_k alpha_t(k) * mu_k[0]

    where mu_k[0] is the mean return in state k (first element of mu_k).

    Signal = clip(E[return] / cost_threshold, -1, 1)

  Method B — Transition-based (more predictive):
    P(Z_{t+1} = j | X_{1:t}) = sum_k alpha_t(k) * A[k,j]

    E[return at t+1] = sum_j P(Z_{t+1} = j) * mu_j[0]

    This uses TOMORROW's expected regime to predict returns, incorporating
    transition dynamics. If the current regime typically transitions to a
    high-return regime, the signal is positive even if the current regime
    is neutral.

  Recommended: Method B (transition-based).

  Numerical example:
    K=3 states. alpha_t = [0.1, 0.7, 0.2] (mostly in state 1).
    mu = [+3 bps, 0 bps, -5 bps] (state returns).
    A = [[0.8, 0.1, 0.1],    (state 0 sticky)
         [0.05, 0.8, 0.15],  (state 1 may go to state 2)
         [0.1, 0.2, 0.7]]    (state 2 sticky)

    P(Z_{t+1}) = alpha_t @ A = [0.1*0.8 + 0.7*0.05 + 0.2*0.1,
                                  0.1*0.1 + 0.7*0.8 + 0.2*0.2,
                                  0.1*0.1 + 0.7*0.15 + 0.2*0.7]
               = [0.135, 0.61, 0.255]

    E[return t+1] = 0.135*3 + 0.61*0 + 0.255*(-5)
                  = 0.405 + 0 - 1.275 = -0.87 bps

    signal = clip(-0.87 / 2.0, -1, 1) = -0.435  (short signal)

    Interpretation: even though we're mostly in state 1 (neutral returns),
    the 15% probability of transitioning to state 2 (negative returns)
    and the 25% chance of already being in state 2 pulls the expected
    return negative.

================================================================================
8.6  MODEL SELECTION
================================================================================

  Number of states K:
    Fit HMM with K = 2, 3, 4, 5, 6.
    Select K minimizing BIC:

      BIC = -2 * log L + n_params * log(T)

    where log L is the log-likelihood from the forward algorithm.

    Expected: K = 3 or 4.
      K=2: bull/bear (too coarse, misses ranging)
      K=3: bull/bear/range (likely best)
      K=4: bull/bear/range/volatile (if volatility is a separate regime)
      K=5+: diminishing returns, overfitting risk

  Covariance type:
    "diag": fewer parameters, assumes feature independence per state.
    "full": captures within-state correlations, but more parameters.
    "tied": shared covariance, fewest params but forced structure.

    Start with "diag". If BIC improves with "full", use "full".

  Regime interpretability check:
    After fitting, examine emission means mu_k:
      State with highest mu_k[0] (return) = "bull"
      State with lowest mu_k[0] = "bear"
      State with mu_k[0] ~ 0 and highest mu_k[3] (vol) = "volatile"
    If states don't map to economically meaningful regimes, the HMM is
    learning noise.

8.8  VALIDATION PROTOCOL
================================================================================

  Phase 1 — Regime interpretability:
    Print emission means per state. Verify that at least one state has
    clearly positive returns and at least one has negative returns.

  Phase 2 — Transition matrix stability:
    Train on days 1-7 and days 8-14. Compare transition matrices.
    Frobenius norm of difference: ||A_1 - A_2|| < 0.3.
    If > 0.3: transition structure is unstable, increase training data.

  Phase 3 — In-sample log-likelihood:
    Verify log L / T > -5.0 (reasonable per-bar log-likelihood).
    If much lower: emission model is a poor fit, check features.

  Phase 4 — OOS signal quality:
    Compute IC between alg_hmm_signal and forward return on held-out data.
    Success: IC > 0.02, positive net bps.

  Phase 5 — Gauntlet:
    nat gauntlet run --last 7 --algorithm hmm_emissions
    Success: Sharpe > 0.5 on >= 2/3 symbols.

8.9  RISK ANALYSIS
================================================================================

  Overfitting risk: MEDIUM-HIGH
    132 parameters with 2000-4000 training samples is tight. The EM
    algorithm can converge to local optima.
    Mitigation: warm-start from GMM, diagonal covariance, minimum
    training bars, BIC for model selection.

  Local optima risk: MEDIUM
    Baum-Welch is EM, which converges to local optima. Different
    initializations can produce different models.
    Mitigation: run 5 random restarts, select model with highest
    log-likelihood. Use GMM warm-start as one of the 5.

  Interpretability risk: LOW-MEDIUM
    HMM states are not labeled — they're just numbers. The states may
    not correspond to economically meaningful regimes.
    Mitigation: post-hoc labeling based on emission means.

  9. STACKING WITH REGIME META-FEATURES

================================================================================

Status:       PROPOSED
Created:      2026-04-04 (design), 2026-06-03 (expanded)
File:         scripts/algorithms/stacking_ensemble.py
Signal:       alg_stack_signal
Polarity:     high_long
ML Method:    Two-level stacking (Ridge Regression or small LightGBM)
Horizon:      5min to 1h
Prerequisite: Multiple base models deployed (#1-#8 + existing algorithms)

================================================================================
9.1  THESIS
================================================================================

Each algorithm captures a different aspect of the market:
  jump_detector    -> post-jump reversion (seconds)
  optimal_entry    -> entry timing via SPRT (seconds)
  funding_reversion -> crowd positioning (hours)
  surprise_signal   -> entropy regime shifts (minutes)

These signals are weakly correlated (Spearman r < 0.3 between most pairs).
A stacking ensemble learns the OPTIMAL WEIGHTING of these signals as a
function of market state — effectively answering "which algorithm should
I trust right now?"

The stacking model is NOT just an average of base signals. It learns
CONDITIONAL weights: in a trending regime, weight jump_detector and
momentum_continuation highly; in a ranging regime, weight mean_reversion
and funding_reversion. The regime meta-features provide this conditioning.

The expected improvement is from DIVERSIFICATION: if base models are
uncorrelated and individually Sharpe ~1.0, a well-weighted portfolio of
N models has theoretical Sharpe ~sqrt(N). With 6 base models, this means
Sharpe ~2.4 for the ensemble.

================================================================================
9.2  ARCHITECTURE
================================================================================

  Level 0 — Base model signals:

    Signal                              Source
    -----------------------------------------------------------------------
    alg_post_jump_reversion             jump_detector (existing)
    alg_entry_signal                    optimal_entry (existing)
    alg_funding_signal                  funding_reversion (existing)
    alg_entropy_surprise                surprise_signal (existing)
    alg_mc_signal                       momentum_continuation (#1)
    alg_mr_signal                       mean_reversion_detector (#2)
    alg_knn_signal                      knn_retrieval (#6)
    alg_rlgbm_signal                    regime_conditioned_lgbm (#7)

    Total: 8 base signals (4 existing + 4 new ML).

  Level 1 — Regime meta-features:

    Feature                             Source
    -----------------------------------------------------------------------
    regime                              GMM label (0-4)
    regime_confidence                   GMM confidence [0,1]
    regime_duration                     Bars since last regime change
    alg_rsm_transition_risk             Regime State Machine (#4)
    ent_tick_1m_slope                   Entropy trend (from bar aggregation)
    cross_model_agreement               Fraction of base signals with same sign:
      = count(sign(base_i) == sign(median(base))) / N_base

    Total: 6 meta-features.

  Level 2 — Stacking model:

    Input: [8 base signals, 6 meta-features] = 14 dimensions
    Target: forward_return (regression) or sign(forward_return) (classification)
    Model: Ridge Regression (primary) or small LightGBM (alternative)

    Ridge Regression (recommended):
      signal = beta_0 + sum_j beta_j * base_signal_j + sum_k gamma_k * meta_k

      Loss: MSE + lambda * ||beta||_2^2
      Regularization: lambda = 1.0 (strong, to prevent overfitting)

      The beta_j coefficients are the "trust weights" for each base model.
      The gamma_k coefficients capture regime-conditional trust adjustments.

    LightGBM stacking (alternative):
      If non-linear interactions between base signals and meta-features
      are important (e.g., "trust jump_detector more when entropy is low"),
      use a small LightGBM:
        max_depth=3, num_leaves=8, n_estimators=50 (very shallow)

================================================================================
9.3  ANTI-LEAKAGE TRAINING PROTOCOL
================================================================================

  CRITICAL: Base model predictions used as Level 0 features must be
  OUT-OF-FOLD to prevent information leakage.

  Procedure:

    1. Split data into K=5 folds (chronological, not random).
    2. For each fold i (i = 1, ..., 5):
       a) For each base model that has trainable parameters (#1, #2, #7, #8):
          - Train on folds {1..5} \ {i} (excluding fold i)
          - Generate predictions on fold i
       b) For non-ML base models (jump_detector, optimal_entry, etc.):
          - Generate predictions on fold i directly (no training needed)
    3. Concatenate all out-of-fold predictions -> Level 0 matrix (N x 8)
    4. Compute meta-features for all bars -> Level 1 matrix (N x 6)
    5. Concatenate: X_stack = [Level_0 | Level_1]  (N x 14)
    6. Train stacking model on X_stack with forward returns as target.

  Embargo between folds: 100 bars (500 min) to prevent label leakage
  from overlapping forward return windows.

  Walk-forward for stacking model:
    Train stacking model on first 80% of out-of-fold predictions.
    Validate on last 20%.
    Success: OOS/IS Sharpe ratio > 0.7 for the stacking model itself.

9.6  VALIDATION PROTOCOL
================================================================================

  Phase 1 — Stacking vs best single model:
    Compare stacking OOS Sharpe vs best single base model OOS Sharpe.
    Success: stacking Sharpe > best single model Sharpe.
    If stacking Sharpe < best single model: the ensemble is overfitting
    or base models are too correlated.

  Phase 2 — Weight stability:
    Train stacking model on weeks 1-2 and weeks 3-4 separately.
    Compare Ridge coefficients. Success: rank order of top-3 weights
    is the same in both periods. If weights flip sign: unstable.

  Phase 3 — Ablation:
    Remove one base model at a time and retrain stacking.
    If removing a model improves OOS Sharpe: that model adds noise.
    Keep only models that decrease Sharpe when removed.

  Phase 4 — Gauntlet:
    nat gauntlet run --last 7 --algorithm stacking_ensemble
    Success: Sharpe > max(individual model Sharpes).

9.7  RISK ANALYSIS
================================================================================

  Overfitting risk: HIGH
    14 input features (8 base + 6 meta) with stacking is high-risk.
    Mitigation: Ridge with strong regularization (alpha=1.0), out-of-fold
    training, embargo, weight stability monitoring.

  Base model correlation risk: MEDIUM
    If base models are too correlated (r > 0.5), the ensemble gains little
    from diversification and may overfit to shared noise.
    Mitigation: check pairwise Spearman before including in stack.
    Exclude models with r > 0.5 with any existing model in the stack.

  Complexity risk: HIGH
    Requires all base models to be deployed and producing predictions.
    Any base model failure (bug, decay, NaN) breaks the entire stack.
    Mitigation: NaN handling (if any base signal is NaN, use global model
    or skip that bar). Monitoring per-model health.

================================================================================
10.1  THESIS
================================================================================

Markets change. The statistical relationships between features and forward
returns shift as participants adapt, as regulation evolves, and as market
microstructure changes. A model trained on January data may be useless by
March.

All ML algorithms in this catalogue (#1-#9) are trained once and deployed
statically until the next retraining cycle. Between retrains, the model
cannot adapt to structural changes. If a regime shift occurs on Monday and
the retrain is on Sunday, the model runs on stale parameters for 6 days.

Online incremental learning solves this with a hybrid approach:
  1. Nightly batch retrain (stable, from-scratch parameter estimation)
  2. Intra-day online SGD updates (responsive, fast adaptation)
  3. Divergence detection (safety net: reset to batch if online drifts)

Existing infrastructure:
  WelfordNormalizer in scripts/algorithms/cascade_probability.py implements
  online mean/variance tracking. The cascade_probability algorithm already
  uses delayed SGD logistic regression — this algorithm generalizes that
  pattern to wrap any base ML model.

================================================================================
10.2  ONLINE SGD UPDATE RULE
================================================================================

  For a logistic regression model with weights beta:

  At each bar t (after forward return r_{t-h} is observed for bar t-h):

  Step 1 — Compute gradient:
    z = beta^T * x_{t-h}
    p = sigma(z) = 1 / (1 + exp(-z))
    y = 1 if r_{t-h} > 0 else 0

    gradient = (p - y) * x_{t-h} + lambda * beta

    (binary cross-entropy gradient + L2 regularization)

  Step 2 — Update weights:
    beta <- beta - eta_t * gradient

  Step 3 — Decay learning rate:
    eta_t = eta_0 / (1 + decay * t)

    where:
      eta_0 = initial learning rate (default 0.01)
      decay = learning rate decay (default 0.001)
      t = number of online updates since last batch retrain

    At t=0 (just after batch retrain):   eta = 0.01
    At t=100:                             eta = 0.01 / 1.1 = 0.0091
    At t=1000:                            eta = 0.01 / 2.0 = 0.005
    At t=5000 (~1 day at 5min bars):     eta = 0.01 / 6.0 = 0.0017

    The decaying rate ensures large updates early (fast adaptation)
    and small updates later (stability).

  Step 4 — Online feature normalization (Welford):
    Before computing z, normalize x_{t-h} using WelfordNormalizer:

      normalizer.update(x_{t-h})
      x_normalized = normalizer.normalize(x_{t-h})

    This tracks the running mean and variance of features, adapting to
    distributional shifts without storing the full history.

  Numerical example:
    Current beta = [0.5, -0.3, 0.8, 0.1, 0.2, -0.1, 0.4, 0.05, 0.3]
    Observation: x = [0.8, 1.2, -0.5, 0.3, 0.6, -0.2, 0.1, 0.9, -0.1]
    True outcome: y = 1 (price went up)

    z = beta^T * x = 0.5*0.8 + (-0.3)*1.2 + 0.8*(-0.5) + ... = 0.42
    p = sigma(0.42) = 0.603
    gradient = (0.603 - 1.0) * x + 0.001 * beta
             = -0.397 * [0.8, 1.2, ...] + 0.001 * [0.5, -0.3, ...]
             = [-0.317, -0.477, 0.199, ...]

    eta_100 = 0.01 / 1.1 = 0.0091
    beta_new = beta - 0.0091 * gradient
             = [0.5 - 0.0091*(-0.317), -0.3 - 0.0091*(-0.477), ...]
             = [0.503, -0.296, ...]

    The weight for the first feature increased from 0.500 to 0.503 because
    the model under-predicted (p=0.603 < y=1.0) and the first feature was
    positive (pointing in the right direction).

10.3  HYBRID BATCH + ONLINE PROTOCOL
================================================================================

  Schedule:

    23:00 UTC — Nightly batch retrain:
      1. Load last 14 days of data
      2. Retrain base model from scratch (full training pipeline)
      3. Evaluate on last 2 days as validation
      4. If validation passes: deploy new model, reset online state
      5. If validation fails: keep previous model, log warning

    Intra-day — Online SGD updates:
      At each bar where forward return becomes available (h bars ago):
        1. Compute SGD gradient
        2. Update online weights
        3. Check divergence (see below)

    The batch model is the GROUND TRUTH. The online updates are ADJUSTMENTS
    on top of the batch model. If the online model diverges too far from
    the batch model, it is reset.

  Divergence detection:

    divergence = ||beta_online - beta_batch|| / ||beta_batch||

    If divergence > max_divergence (default 0.5):
      Reset: beta_online <- beta_batch
      Log warning: "Online model diverged, resetting to batch."

    This prevents the online model from drifting to a bad solution due to
    a sequence of noisy updates.

================================================================================
10.4  DECAY DETECTION
================================================================================

  Three metrics to detect model decay:

  Metric 1 — Rolling Sharpe:
    Compute Sharpe on trailing 7-day and 30-day windows.
    Alarm conditions:
      7-day Sharpe < 0.0 AND 30-day Sharpe < 0.3 -> model decaying
      7-day Sharpe < -0.5                         -> immediate shutdown

  Metric 2 — Rolling IC (Information Coefficient):
    IC = Spearman(model prediction, realized return) over 7-day window.
    Alarm: IC < 0.005 for > 48 hours.

  Metric 3 — Feature importance drift:
    For LightGBM models: compare feature importance ranking between
    current and previous retrain. If Kendall tau < 0.5 -> structural shift.
    For logistic regression: compare coefficient signs. If any top-3
    coefficient flips sign -> structural shift.

  Actions on decay detection:
    1. Reduce position size by 50% immediately
    2. Trigger emergency retrain with shorter window (7 days)
    3. If emergency retrain fails: disable model entirely
    4. Alert via Telegram (if configured)
    5. Fall back to non-ML algorithms (which have no learned parameters)

10.7  VALIDATION PROTOCOL
================================================================================

  Phase 1 — Convergence test:
    On synthetic data with known change-points, verify:
    - Online model converges to batch model when no change (eta decays)
    - Online model adapts when distribution shifts (gradient kicks in)
    - Divergence detection fires and resets correctly

  Phase 2 — Online vs batch comparison:
    Run both models on same OOS data for 14 days.
    Compare: daily Sharpe of online vs batch.
    Success: online Sharpe >= batch Sharpe on average.

  Phase 3 — Decay detection test:
    Inject a deliberate model degradation (e.g., zero out all weights).
    Verify: decay detection fires within 48 hours.
    Verify: emergency retrain restores performance.

  Phase 4 — Long-run stability:
    Run for 30 days of OOS data. Monitor:
    - No single 7-day window with Sharpe < -0.5
    - Model uptime > 90% (not disabled by decay > 10% of time)

================================================================================
10.8  RISK ANALYSIS
================================================================================

  Gradient noise risk: MEDIUM
    Individual SGD updates are noisy. A sequence of bad updates can push
    weights to a poor region.
    Mitigation: learning rate decay, L2 regularization, divergence reset.

  Catastrophic forgetting risk: LOW
    The nightly batch retrain anchors the model. Online updates are small
    perturbations. The max_divergence check prevents the online model from
    straying too far.

  Complexity risk: MEDIUM
    Two models running in parallel (batch and online), plus divergence
    monitoring, decay detection, and emergency retrain logic.
    Mitigation: extensive logging, Telegram alerts, automatic fallback.

  Latency risk: LOW
    One SGD update per bar (9 multiplications + 9 additions) = ~100ns.
    Normalizer update: ~100ns. Total: negligible.

A.1  BAR AGGREGATION PIPELINE
--------------------------------------------------------------------------------

All ML algorithms operate on bar-aggregated data, not raw 100ms ticks.
Aggregation is performed by aggregate_bars() in
scripts/cluster_pipeline/preprocess.py.

Default timeframe: 5min (288 bars/day per symbol).

Per-feature aggregation rules:
  Most features:      mean, std, last
  Price columns:      open, high, low, close, mean
  Flow volumes:       sum
  Entropy (ent_*):    mean, std, slope (OLS trend over bar)
  Whale flow:         sum

Feature naming: {original}_{suffix} (e.g., ent_tick_1m_mean, raw_midprice_close)

        Complexity: ~200 LOC.
        Value: adaptive, non-parametric, complementary to ML models.
        Dependency: bar aggregation + database management.

    #7  Regime LightGBM        Multiple models, limited per-regime data.
        Complexity: ~400 LOC + 6 training scripts.
        Value: uncertain until #4 proves regime labels are real.
        Dependency: #4 (regime labels).

    #8  HMM Emissions          Most complex single algorithm.
        Complexity: ~350 LOC.
        Value: high if regimes are real, hard to validate.
        Dependency: hmmlearn, sufficient training data.

        Complexity: ~300 LOC + anti-leakage protocol.
        Value: diversification, but only if base models are diverse.
        Dependency: #1, #2, #6, #7 deployed.

    #10 Online Learning        Infrastructure, not an algorithm.
        Complexity: ~250 LOC.
        Value: only after static models prove edge.
        Dependency: any ML base model deployed.

    are not applicable.

================================================================================
D.8  CONTINUOUS MONITORING TESTS (post-deployment)
================================================================================

  Once an ML algorithm is deployed, ongoing tests run nightly as part of
  the research agent cycle (see agent_specs_1.md).

  -----------------------------------------------------------------------
  Model health checks (nightly at 23:00 UTC)
  -----------------------------------------------------------------------

  check_model_age
      If model timestamp > 14 days: WARNING.
      If model timestamp > 30 days: CRITICAL. Trigger retrain.

  check_ic_rolling
      Rolling 7-day IC (Spearman of signal vs realized return).
      If IC < 0.005 for > 48 hours: WARNING.
      If IC < 0 for > 72 hours: CRITICAL. Disable algorithm.

  check_sharpe_rolling
      Rolling 30-day Sharpe from simulated PnL.
      If Sharpe < 0 for > 7 days: WARNING.
      If Sharpe < −0.5 (any window): CRITICAL. Immediate disable.

  check_feature_importance_drift (#1, #2, #7 only)
      Compare current LightGBM feature importance ranking to previous
      retrain. Kendall tau < 0.5: WARNING (structural shift).

  check_online_divergence (#10 only)
      If ||w_online - w_batch|| / ||w_batch|| > max_divergence:
      Auto-reset online weights. Log event.

  check_regime_distribution (#4, #7, #8 only)
      If any regime occupancy drops to 0% for > 24 hours:
      Regime model may have collapsed. Trigger re-fit.

  -----------------------------------------------------------------------
  Alert routing
  -----------------------------------------------------------------------

  WARNING:  Log to data/research/alerts.json.
            Include in nightly Telegram report.
  CRITICAL: Immediate Telegram alert.
            Disable algorithm in ALGO_CONFIG.
            Flag for manual review.

This appendix maps each of the 10 ML algorithms to the existing codebase

================================================================================
E.1  EXISTING ARCHITECTURE SUMMARY
================================================================================

  The algorithm framework has 5 layers. ML algorithms must integrate with
  all 5 to be production-ready.

  Layer 1 — Algorithm code (scripts/algorithms/<name>.py)
    Implements MicrostructureAlgorithm ABC.
    Decorated with @register from registry.py.
    Auto-discovered by autodiscover.py at import time.

  Layer 2 — Configuration (config/algorithms.toml)
    Per-algorithm TOML section with tunable parameters.
    Constructor kwargs map 1:1 to TOML keys.

  Layer 3 — Paper trading (scripts/alpha/paper_trader_generic.py)
    ALGO_CONFIG dict: primary signal, polarity, bar aggregation.
    Walk-forward z-score calibration on prior 3 days.
    100-minute fixed horizon trades.

  Layer 4 — Gauntlet (scripts/alpha/overnight_sweep.py)
    DAILY_ALGOS list: all algorithms evaluated in multi-day sweep.
    Per-symbol per-date PnL aggregation.

  Layer 5 — Agent (scripts/agent/runner.py)
    5-gate hypothesis protocol: IC, cost, temporal, symbol, correlation.
    Automatic signal registration via signal_adapter.py.

  Current state: 18 hand-coded algorithms, all statistical/signal-processing.
  Zero ML algorithms deployed. cascade_probability is the only existing
  algorithm with inline online learning (SGD), but it is self-contained
  and does not use the model_io or walk_forward infrastructure.

  -----------------------------------------------------------------------

  -----------------------------------------------------------------------

  Algorithm file structure:

    class MomentumContinuation(MicrostructureAlgorithm):
        def __init__(self, model_path="models/momentum_continuation", ...):
            self._model, self._scaler, self._meta = None, None, None
            try:
                path, meta = model_io.get_latest_model(model_path)
                self._model, self._scaler, _ = model_io.load_sklearn_model(path)
            except FileNotFoundError:
                pass  # step() will return NaN

        def step(self, tick):
            if self._model is None:
                return {f.name: np.nan for f in self.alg_features()}
            x = self._extract_features(tick)
            if np.any(np.isnan(x)):
                return {f.name: np.nan for f in self.alg_features()}
            x_scaled = self._scaler.transform(x.reshape(1, -1))
            prob = self._model.predict_proba(x_scaled)[0, 1]
            if prob > self._p_long:
                signal = (prob - 0.5) * 2.0
            elif prob < self._p_short:
                signal = -(0.5 - prob) * 2.0
            else:
                signal = 0.0
            gated = signal if tick.get("ent_tick_1m_mean", 1.0) < self._entropy_threshold else 0.0
            return {"alg_mc_signal": gated,
                    "alg_mc_probability": prob,
                    "alg_mc_entropy_gate": 1.0 if gated != 0.0 or signal == 0.0 else 0.0}

  Training script pattern:
    1. load_features(symbols=["BTC","ETH","SOL"], date_range=(...))
    2. aggregate_bars(df, "5min")
    3. Compute forward returns (6-bar = 30min horizon)
    4. Binary label: return > 0 → 1, else 0
    5. Train LogisticRegression(C=1.0, penalty='l2')
    6. walk_forward_validation(n_splits=5, embargo_bars=6)
    7. model_io.save_sklearn_model(model, scaler, metadata, "models/momentum_continuation")

  ALGO_CONFIG entry:
    "momentum_continuation": {
        "primary": "alg_momentum_signal",
        "polarity": "high_long",
        "bar_agg": "mean",
    }

  Key design decision: the algorithm loads a pre-trained model at __init__
  and uses it for inference only. Training is a separate offline step.
  This matches the pattern used by all other stateful algorithms (Kalman
  filters load OU parameters, Hawkes loads decay estimates) — the
  difference is that ML algorithms load learned weights instead of
  calibrated parameters.

        def __init__(self, base_algorithm="jump_detector", ...):
            self._base = get_algorithm(base_algorithm)
            self._meta_model = load_meta_model(base_algorithm)

        def step(self, tick):
            base_out = self._base.step(tick)
            base_signal = base_out[self._base_primary]
            meta_features = self._extract_meta_features(tick, base_out)
            prob = self._meta_model.predict_proba(meta_features)[0, 1]
            signal = base_signal if prob > self._threshold else 0.0
            return {
                "alg_meta_signal": signal,
                "alg_meta_probability": prob,
                "alg_meta_base_raw": base_signal,
                "alg_meta_active": 1.0 if abs(base_signal) > 0.01 else 0.0,

  Files to CREATE:

  Files to MODIFY:

  Infrastructure REUSED:
    aggregate_bars()                    bar aggregation
    (Ledoit-Wolf from sklearn.covariance already available)

  Key integration: ONLINE DATABASE, no offline training.

    Unlike all other ML algorithms, KNN builds its database at runtime
    as new bars arrive. The run_batch() call processes bars sequentially,
    storing each (feature_vector, forward_return) pair in a circular buffer.

    This makes it similar to cascade_probability.py in architecture: both
    maintain internal state that evolves with data. The difference is that
    KNN stores raw instances while cascade_probability stores learned weights.

    The Mahalanobis distance requires a covariance estimate. Options:
      (a) Use sklearn.covariance.LedoitWolf() on the first N bars
          (already available in the environment — imported in multiple scripts)
      (b) Update covariance incrementally using Welford-like online update
          (more complex but avoids O(N) recomputation)

    Recommendation: (a) for initial implementation. Recompute covariance
    every 1000 bars (configurable). The O(d²) recomputation on 7 features
    is negligible compared to the O(N·d) KNN search.

    [knn_retrieval]
    k = 20
    max_history = 5000
    cov_update_interval = 1000
    time_decay_halflife = 500          # bars

E.8  BAR VS TICK: EXECUTION MODE ANALYSIS
================================================================================

  All existing algorithms operate on RAW TICKS (100ms). ML algorithms
  operate on AGGREGATED BARS (5min). This creates an architectural gap.

  Current pipeline for hand-coded algorithms:
    parquet (ticks) → algo.run_batch(df) → features (tick-level)
    → aggregate to bars → paper trader → trades

  Required pipeline for ML algorithms:
    parquet (ticks) → aggregate_bars() → bars
    → algo.run_batch(bars) → features (bar-level)
    → paper trader → trades

  The difference: ML algorithms receive pre-aggregated bars, not raw ticks.
  This means:

  1. run_batch() expects bar-level DataFrame columns (with _mean, _std,
     _slope suffixes), NOT raw tick columns.

  2. required_columns() must return bar-aggregated names:
       ["ent_tick_1m_mean", "trend_hurst_300_mean", ...]
     NOT raw tick names:
       ["ent_tick_1m", "trend_hurst_300", ...]

  3. step() operates on one BAR (not one tick). The dict passed to step()
     has bar-aggregated values.

  4. The paper trader already handles this: it aggregates ticks to 5min bars
     before computing z-scores. But AlgorithmRunner.run_on_parquet() loads
     raw ticks and passes them to run_batch().

  Resolution: ML algorithms implement run_batch() with internal aggregation:

    def run_batch(self, df: pd.DataFrame) -> pd.DataFrame:
        # If input is tick-level (has timestamp_ns, no _mean suffix):
        if "timestamp_ns" in df.columns:
            bars = aggregate_bars(df, timeframe="5min")
        else:
            bars = df  # already aggregated
        # Run inference on bars
        ...
        # Expand back to tick-level by forward-filling
        return tick_level_result

  This preserves compatibility with AlgorithmRunner.run_on_parquet()
  (which passes raw ticks) and with paper_trader_generic.py (which
  passes bars). The output is always aligned to the input DataFrame's
  index.

  Alternative: modify AlgorithmRunner to detect bar-level algorithms
  and pre-aggregate. This is cleaner but requires changes to runner.py.

  Recommendation: implement internal aggregation in each ML algorithm's
  run_batch() for now (no framework changes). Refactor to runner.py
  after 3+ ML algorithms are deployed and the pattern is proven.
