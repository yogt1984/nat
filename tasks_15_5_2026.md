## Phase 1: Gate Hardening (in progress)

### 1.1 Gate-specific IC extraction [DONE]
- Parse `single_factors` array in regime screener JSON for the exact `regime_gate` label
- Extract `ic_filt_5s` for the specific gate, not the aggregate report IC
- Fallback to aggregate IC for non-regime reports (profile scalp)

### 1.2 dIC gate [DONE]
- Require `gated_IC - baseline_IC >= min_dIC` (default 0.05)
- The gate must add value over the ungated signal
- Added to `check_dIC_gate()` in `runner.py`

### 1.3 Cost gate [DONE]
- After discovery passes, run `nat spannung backtest --symbol BTC --data {data_dir}`
- Parse `avg_return_per_trade_bps` from backtest best threshold level
- Gate: `avg_return_per_trade_bps >= 0.1` (gross signal edge check, not net-of-costs)
- Failure reason: `cost_killed` (eligible for recycler re-evaluation)
- Note: net_sharpe_maker is too strict (-335 for all signals due to 1bps/side maker cost vs 0.4bps avg return); gross edge is the right filter at this stage

### 1.4 Correlation deduplication [DONE]
- Before registering a new signal, compute max Spearman correlation with all existing registry signals
- Reject if `max_corr > 0.7` — the signal is redundant
- Implementation: load Parquet data, apply regime gate mask, compute pairwise Spearman rho on gated values
- Failure reason: `redundant`
- Result: l5 (rho=0.948) and l10 (rho=1.000) correctly blocked vs l1; registry: 3 → 1 signal

### 1.5 FDR control (Benjamini-Hochberg) [DONE]
- IC p-value computed via z-test: z = IC * sqrt(n_obs), two-sided
- P-values stored in gate result messages during discovery
- BH procedure applied at end of each cycle at q=0.05 (configurable in agent.toml)
- Hypotheses that don't survive FDR: status=failed, reason=fdr_rejected, removed from registry
- Current data: all p-values ≈ 0 (IC>0.45, n=28800) — no rejections yet; gate activates on marginal signals

## Phase 2: Agent Intelligence

### 2.1 Adaptive IC threshold
- As registry grows, raise the bar: `min_ic = max(0.10, median(registry_ic) * 0.8)`
- Prevents registry from filling with marginal signals
- Update at start of each cycle based on current registry state

### 2.2 IC decay monitoring
- For each registered signal, track rolling 7-day IC on new data
- If IC drops below 50% of discovery IC for 14 consecutive days, auto-retire
- Add `status=retired` and `retired_reason` to RegisteredSignal
- New agent phase: MONITOR (runs after EXECUTE)

### 2.3 Computation caching [DONE]
- `nat spannung regime` on the same (date, symbol) produces identical output
- Cache report JSONs keyed by `hash(command, date, symbol)` in `data/agent/cache/`
- TTL-based expiry (7 days default), corruption-resilient, flag-order-invariant keys
- Measured: 85% hit rate, cycle time 5 min → 2 min 13s (56% faster)
- 57 unit + integration tests in `test_agent_cache.py`

### 2.4 Hypothesis chaining
- When a hypothesis passes discovery but fails symbol replication on exactly 1 symbol:
  - Spawn a symbol-specific variant with lower priority
  - Track as `parent_id` chain
- When a hypothesis passes with dIC >> min_dIC:
  - Spawn ensemble variants testing 2-gate combinations

### 2.5 Ensemble gate testing
- Generator: `generators/ensemble.py`
- For each pair of passing single gates, test their intersection: `gate_A AND gate_B`
- Expected: higher IC at lower coverage
- Priority: boost for uncorrelated gate pairs (entropy x illiquidity)

## Phase 4: Multi-Agent Architecture

### 4.1 Agent base class refactor
- Extract abstract `ResearchAgent` interface from current `AgentDaemon`
- Methods: `generate()`, `execute()`, `get_tools()`, `frequency_band`, `horizon_range`
- Current logic becomes `MicrostructureAgent(ResearchAgent)`
- Shared: `Hypothesis`, `RegisteredSignal`, `HypothesisQueue`, `manifest.py`
- Agent-specific: generators, gate checks, nat command invocations

### 4.2 Medium-frequency agent
- Horizon: 1min - 1h
- Signals: momentum (trend slope, R2), vol breakout (vol_ratio > 2), flow clustering
- Gates: HMM state, time-of-day (UTC session), funding rate regime
- Tools: `nat profile scalp --timeframe 5min`, `nat cluster hmm-fit`, `nat alpha validate`
- Data: same Parquet files, resampled to 1min/5min bars
- Value: signals at this horizon are uncorrelated with 5s microstructure signals

### 4.3 Macro agent
- Horizon: 1h - 24h
- Signals: funding rate mean-reversion (8h cycle), OI divergence, whale flow momentum
- Gates: funding z-score regime, vol regime (1d), BTC dominance
- Tools: `nat macro`, `nat alpha validate --timeframe 1h`
- Data: context features resampled to 1h bars
- Requires: external data sources (cross-exchange prices, TradFi correlation)

### 4.4 Meta-agent (orchestrator)
- Allocates compute budget across agents (Thompson bandit at agent level)
- Nightly correlation monitoring across all agents' signals
- Portfolio assembly: risk parity weighting of uncorrelated signals
- Promotion decisions: portfolio-level Sharpe, not individual signal Sharpe

## Phase 5: New Features (Rust Ingestor)

### 5.1 Order book resilience
- Time to recover 50% of depth after a large take
- Per-event feature, requires trade-triggered measurement
- Reference: Biais, Hillion & Spatt (1995)

### 5.2 OBI dynamics
- `d(OBI_l5)/dt` — imbalance velocity (1s finite difference)
- `d²(OBI_l5)/dt²` — imbalance acceleration
- Inflection points may predict direction changes before level shifts

### 5.3 Hawkes trade intensity
- `lambda(t) = mu + sum(alpha * exp(-beta * (t - t_i)))` over 30s window
- Self-exciting arrival: clusters of trades predict more trades
- Reference: Bacry, Mastromatteo & Muzy (2015)

### 5.4 Cross-symbol imbalance
- `OBI_BTC(t) - OBI_ETH(t)` — relative pressure
- Requires synchronized feature vectors across symbols
- Implementation: shared state between symbol tasks via `Arc<RwLock<>>`

### 5.5 Queue position proxy
- `depth_at_touch / avg_trade_size` — estimated time-to-fill for maker orders
- Directly relevant for market-making execution

### 5.6 Funding rate momentum
- `funding(t) - funding(t - 8h)` — Hyperliquid 8h settlement cycle
- Higher frequency than raw funding; captures mean-reversion timing
- Requires 8h lookback buffer in context features
